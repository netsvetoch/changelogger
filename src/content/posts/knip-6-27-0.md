Вышел минорный релиз [Knip 6.27.0](https://github.com/webpro-nl/knip/releases/tag/knip%406.27.0) — инструмента для поиска неиспользуемых файлов, зависимостей и exports в JavaScript- и TypeScript-проектах. В этом выпуске появились новый plugin для Temporal, дополнительные точки входа для TanStack Start, поддержка Vue 2-шаблонов и несколько исправлений, которые делают граф модулей точнее в реальных monorepo и script-конфигурациях.

Источники: [GitHub Release](https://github.com/webpro-nl/knip/releases/tag/knip%406.27.0), [compare 6.26.0…6.27.0](https://github.com/webpro-nl/knip/compare/knip%406.26.0...knip%406.27.0), а также коммиты [98191a0](https://github.com/webpro-nl/knip/commit/98191a04aad0572bead191b101bfecef75067b0c), [2157f39](https://github.com/webpro-nl/knip/commit/2157f3913f77216d268b2255899b6fc58609f9bb), [57a4770](https://github.com/webpro-nl/knip/commit/57a4770447cc4c8f9ea4ffb9171c0c92eed4f255), [1b0a2c0](https://github.com/webpro-nl/knip/commit/1b0a2c0ee784db165c7f92fe10397a6cb513c1cf) и [b5a3176](https://github.com/webpro-nl/knip/commit/b5a31767a83e2cec3fb20c3b146d94c0412a4fd6). Релиз имеет формат `6.27.0`, поэтому это qualifying minor-релиз; `featured` оставлен `false`.

## Новый plugin для Temporal

Knip получил встроенный `temporal` plugin. Он активируется, когда в проекте есть `@temporalio/worker`, и регистрирует workflow-файлы как production entries. По умолчанию учитываются `src/workflows` и `src/workflows/index` с расширениями `.js`, `.cjs`, `.mjs`, `.ts`, `.cts` и `.mts`.

Это закрывает типичную особенность Temporal: workflow запускаются runtime через путь к каталогу и не обязательно имеют обычный статический импорт из entrypoint. Например, такой worker теперь связывает папку workflows с графом проекта:

```ts
import { Worker } from "@temporalio/worker";
import * as activities from "./activities/index.ts";

const worker = await Worker.create({
  workflowsPath: "./src/workflows",
  activities,
});
```

В конфигурации Knip plugin можно переопределить так же, как другие плагины:

```json
{
  "temporal": {
    "entry": ["src/workflows/**/*.ts"]
  }
}
```

Отдельный fixture в исходном репозитории проверяет и обычный, и production-анализ Temporal-проекта: workflow `src/workflows/fulfill-order.ts` и его re-export из `index.ts` попадают в обработанный граф.

## TanStack Start распознаётся как приложение с роутером

`tanstack-router` plugin теперь включается не только для пакетов TanStack Router, но и для `@tanstack/react-start` и `@tanstack/solid-start`. В production entries добавлены:

```text
src/routeTree.gen.{ts,js}
src/{router,start,client,server}.{js,jsx,ts,tsx}
```

Таким образом, проект TanStack Start больше не требует вручную добавлять `src/router.tsx` или `src/start.ts` в `entry`, даже если зависимость приложения — `@tanstack/react-start`, а не отдельный router package.

## Vue 2: анализ auto-imports из template compiler

В предыдущем релизе Knip улучшил поддержку Vue auto-imports для современных SFC. В 6.27.0 тот же сценарий расширен на Vue 2, где `vue/compiler-sfc` предоставляет другой API: `parseComponent` и компиляцию шаблона в render function.

Изменение в `packages/knip/src/plugins/_vue/auto-import.ts` делает три вещи:

- определяет Vue 2 по наличию `parseComponent`;
- вызывает `compileTemplate` с параметрами `sourceMap: false`, `isProduction: true` и `prettify: false`;
- извлекает обращения вида `_vm.formatApple(...)` из сгенерированного render-кода, пропуская внутренние поля и `$set`/`$delete`.

Кроме того, обход Vue 2 AST теперь посещает `ifConditions` и `scopedSlots`. Это важно для компонентов, которые используются только в ветви `v-if`, slot или выражении шаблона.

Например, при наличии сгенерированного `auto-imports.d.ts` и `components.d.ts` Knip теперь учитывает вызовы и компоненты в Vue 2-шаблоне:

```vue
<template>
  <main>
    <span v-if="count">Growing</span>
    <AppleCard v-else :label="formatApple(count)" />
  </main>
</template>
```

В тестовом проекте `AppleCard.vue` и `formatApple.ts` помечаются используемыми, а неиспользуемые `BananaCard.vue` и `formatBanana.ts` остаются в отчёте. То есть поддержка не превращает весь каталог в набор entries — она извлекает конкретные identifiers и теги из шаблона.

## Tailwind CSS: `@import url(...)` стал входом в граф

Tailwind compiler теперь обрабатывает локальные CSS URL-импорты во всех трёх формах:

```css
@import url("./double-quoted.css");
@import url("../single-quoted.css");
@import url(./unquoted.css);
```

Такие импорты преобразуются во внутренние inputs Knip так же, как `@import "./file.css"`, `@config` и `@plugin`. Внешние адреса намеренно пропускаются:

```css
@import url("https://cdn.example.com/theme.css");
@import url(//cdn.example.com/theme.css);
```

Парсер также не считает директивы внутри block comments и строковых CSS-значений реальными импортами. Это уменьшает одновременно две группы ошибок: пропущенные локальные stylesheet dependencies и ложные `unresolved` для закомментированных примеров.

## Unbuilt workspace exports и custom conditions

В monorepo пакет может публиковать исходники через нестандартное условие `exports`, пока `dist` ещё не собран:

```json
{
  "exports": {
    ".": {
      "monorepo": "./src/index.ts",
      "types": "./dist/index.d.ts",
      "import": "./dist/index.js",
      "default": "./dist/index.js"
    },
    "./format/*": {
      "monorepo": "./src/*.format.ts",
      "default": "./dist/*.format.js"
    }
  }
}
```

Knip 6.27.0 теперь умеет сопоставлять точные subpath keys и wildcard-паттерны, выбирая наиболее специфичное совпадение. Если обычное разрешение не находит собранный target, анализатор перебирает conditional targets и берёт первый существующий на диске файл, кроме type-only `.d.ts` targets. Это работает и для `package.json#imports`.

Важная деталь безопасности: target должен начинаться с `./`, а сегменты `..` и `node_modules` отбрасываются. Поэтому fallback не превращается в разрешение пути за пределами workspace.

## Script references: учитываются аргументы после `run`

Логика анализа команд вынесена в общий `expandScript` и теперь расширяет пользовательские scripts с переданными аргументами для Bun, npm, pnpm и Yarn. Например, если `package.json` содержит:

```json
{
  "scripts": {
    "with-env": "node --import tsx"
  }
}
```

то команда:

```bash
pnpm run with-env -- src/main.ts
```

учитывает не только бинарник `node` и `tsx`, но и `src/main.ts` как deferred entry. Аналогично обрабатываются `npm run`, `yarn run`, `bun run` и формы без явного `run` там, где их поддерживает менеджер.

Для защиты от циклических scripts используется `expandedScripts`: если `pnpm loop` ссылается на сам себя, повторного раскрытия не происходит. Для Yarn сохранён и корректный `cwd` при `--cwd`, а для Bun дополнительно разрешаются локальные бинарники, запускаемые через `bun run`.

## Точнее определяется, что является binary и dependency

Несколько изменений в этом выпуске корректируют границы анализа:

- токен бинарника с `/`, например `target/release/mytool`, трактуется как file reference, а не как имя executable dependency; это особенно важно для gitignored каталога `target/`;
- в список известных глобальных бинарников добавлены `claude`, `launchctl`, `pulumi`, `systemctl`, `tmux` и `tmuxinator`, поэтому их не нужно объявлять фиктивными dependencies;
- импорт через alias в `node_modules` атрибутируется объявленному alias-пакету, а не физическому имени найденного каталога;
- unresolved import транзитивной peer dependency теперь помечается как `unlisted`, если пакет не установлен. При этом реально разрешённая транзитивная peer dependency сохраняет прежнее поведение;
- ссылки из `tsconfig.json` на `compilerOptions.types` учитываются в workspace-владельце, а ambient `declare module` в script-файлах больше не считается обычным runtime import;
- зависимости, перечисленные в `tsdown` `neverBundle`, и пакеты из `vite.ssr.external` считаются referenced dependencies.

Для Vite поддерживаются строковые значения `ssr.external`; регулярные выражения остаются частью типа конфигурации, но не могут быть преобразованы в имя пакета:

```ts
export default defineConfig({
  ssr: {
    external: ["native-renderer", "transitive-runtime", /^@grpc\//],
  },
});
```

В результате `native-renderer` и `transitive-runtime` не попадут в отчёт как неиспользуемые или незаявленные только потому, что Vite загружает их во время SSR отдельно от обычного import graph.

## Изменения в React Email и Vitest

React Email plugin теперь понимает короткий alias `-d` для аргумента `--dir`:

```bash
email dev -d ./preview-dir
```

Каталог `preview-dir` анализируется так же, как каталог, переданный через `--dir`.

Vitest получил две небольшие, но практичные доработки:

- в `test.snapshotSerializers` теперь разрешаются локальные файлы и зависимости, аналогично `setupFiles`;
- в inline `test.projects` поддерживается `extends: './vitest.shared.config.ts'`, поэтому shared-конфигурация становится входом, а её `setupFiles` не теряются при анализе.

## Обновление

Установите новую минорную версию и снова запустите проверку проекта:

```bash
pnpm add -D knip@6.27.0
pnpm knip
```

Если проект использует Temporal, TanStack Start, Vue 2 auto-imports или unbuilt workspace exports, сначала проверьте, что встроенный plugin действительно включился и больше не нужны дублирующие ручные `entry`/`ignore`. Для script-команд с аргументами стоит проверить отчёт после обновления: теперь Knip может увидеть дополнительные файлы, переданные в `npm run`, `pnpm run`, `yarn run` или `bun run`.
