[`pi`](https://github.com/earendil-works/pi) выпустил минорный релиз [`v0.82.0`](https://github.com/earendil-works/pi/releases/tag/v0.82.0). Главная техническая тема версии — управление тем, **как модель формирует аргументы tools**: инструмент может запросить строгое соблюдение JSON Schema либо, для поддерживаемых OpenAI-моделей, генерацию по Lark/regex-грамматике. Одновременно CLI получил два OAuth-входа, bash — контекст активной сессии, а RPC-клиенты — потоковые события вывода команд.

Источники: GitHub Release [`earendil-works/pi@v0.82.0`](https://github.com/earendil-works/pi/releases/tag/v0.82.0), compare [`v0.81.0...v0.82.0`](https://github.com/earendil-works/pi/compare/v0.81.0...v0.82.0), документация тега для [constrained sampling](https://github.com/earendil-works/pi/blob/v0.82.0/packages/ai/README.md#constrained-sampling-for-tools), [провайдеров](https://github.com/earendil-works/pi/blob/v0.82.0/packages/coding-agent/docs/providers.md#openrouter), [окружения bash](https://github.com/earendil-works/pi/blob/v0.82.0/packages/coding-agent/docs/environment-variables.md#bash-tool-session-environment) и [RPC](https://github.com/earendil-works/pi/blob/v0.82.0/packages/coding-agent/docs/rpc.md#bash_execution_update), а также PR [#6341](https://github.com/earendil-works/pi/pull/6341), [#6927](https://github.com/earendil-works/pi/pull/6927), [#6935](https://github.com/earendil-works/pi/pull/6935), [#6967](https://github.com/earendil-works/pi/commit/bb3d7d399c06e5fe284f34eb66b15b037ab18649) и [#6971](https://github.com/earendil-works/pi/pull/6971).

## Tool может требовать строгую JSON Schema

В `@earendil-works/pi-ai` у типа `Tool` появилось поле `constrainedSampling`. Конфигурация `{ type: "json_schema", strict: "prefer" }` просит провайдера включить server-side strict mode, но сохраняет обычный function calling как запасной путь. Значение `"require"` принципиально отличается: Pi прекращает запрос с понятной ошибкой, если выбранная пара provider/model не может гарантировать строгий режим.

```ts
import { Type, type Tool } from "@earendil-works/pi-ai";

const editFile: Tool = {
  name: "edit_file",
  description: "Replace a file's content.",
  parameters: Type.Object(
    {
      path: Type.String(),
      content: Type.String(),
    },
    { additionalProperties: false }
  ),
  constrainedSampling: {
    type: "json_schema",
    strict: "require",
  },
};
```

Это не флаг, который Pi безусловно посылает каждому OpenAI-compatible endpoint. Совместимость хранится в metadata модели: для OpenAI Responses используется `supportsStrictMode`, для Anthropic — `supportsStrictTools`; custom model configs также могут объявить эти возможности. Встроенные Anthropic-модели помечены как strict-capable, а для Amazon Bedrock признак строится из metadata structured output.

Поддерживаемый native strict mode охватывает OpenAI, Anthropic, подходящие Bedrock Converse-модели, Mistral и Gemini 3 через Google Generative AI/Vertex. У Google Pi выбирает function-calling mode `VALIDATED` (либо `ANY`, если он явно задан). Более ранние Gemini при `prefer` остаются на обычных tools, а при `require` не проходят запрос: они не обеспечивают required-параметры.

## OpenAI custom tools с Lark и regex-грамматиками

Второй вариант `constrainedSampling` предназначен для случая, когда аргументом инструмента должен быть не произвольный JSON-объект, а текст строго определённого языка — например, patch. В 0.82.0 Pi умеет отправлять OpenAI custom tool с Lark- или regex-грамматикой; если задать оба варианта, приоритет у Lark.

```ts
const applyPatch: Tool = {
  name: "apply_patch",
  description: "Apply a unified diff.",
  parameters: Type.Object(
    { input: Type.String({ description: "Unified diff text" }) },
    { additionalProperties: false }
  ),
  constrainedSampling: {
    type: "grammar",
    variants: {
      openai_lark: String.raw`start: /(.|\n)+/`,
    },
  },
};
```

Контракт здесь намеренно узкий: schema должна быть object с **ровно одним обязательным строковым свойством**. Pi по-прежнему отдаёт автору tool JSON вида `{ input: string }`, но провайдер генерирует саму строку по grammar. Для неподдерживаемой модели grammar tool откатывается к обычному function/JSON-schema tool; пустой набор подходящих grammar variants на grammar-capable модели — это ошибка валидации, а не молчаливый запрос.

Поддержка тоже привязана к каталогу, через `supportsOpenAIGrammarTools`: встроенный каталог включает её для GPT-5+ на OpenAI, OpenAI Codex, Azure OpenAI Responses, GitHub Copilot, OpenCode и Cloudflare AI Gateway. Не все шлюзы сохраняют OpenAI `type: "custom"`: например, OpenRouter нормализует tool schemas, поэтому Pi не заявляет такую возможность для него. Это защищает интеграции от отправки новых полей endpoint'у, который их исказит или отвергнет.

## `/login` для OpenRouter и Kimi Code

OpenRouter теперь можно авторизовать из Pi без ручного добавления `OPENROUTER_API_KEY`:

```text
/login openrouter
```

В меню выбирается **Sign in with OpenRouter**. PR [#6927](https://github.com/earendil-works/pi/pull/6927) реализует browser authorization с PKCE S256 и одноразовым localhost callback: у него есть timeout, отмена и проверка ответа. Результатом становится контролируемый пользователем API key, который работает и для text-, и для image-provider; прежний вариант с `OPENROUTER_API_KEY` остаётся доступным через **Use an API key**.

Для `kimi-coding` добавлена авторизация подписки Kimi Code. Она использует device authorization grant RFC 8628, polling токена и rotation refresh token с retry/backoff; при этом переменная `KIMI_API_KEY` никуда не исчезает. Иными словами, это не замена key-based access, а второй способ завести credentials через `/login`.

## Bash видит активную сессию, RPC — его потоковый вывод

Команды, выполненные LLM-вызовом встроенного bash tool, теперь получают значения, вычисленные на старте каждой команды:

| Переменная           | Значение                                                        |
| -------------------- | --------------------------------------------------------------- |
| `PI_SESSION_ID`      | идентификатор текущей сессии                                    |
| `PI_SESSION_FILE`    | абсолютный путь к JSONL сессии; отсутствует у ephemeral session |
| `PI_PROVIDER`        | выбранный в Pi provider                                         |
| `PI_MODEL`           | выбранный model ID                                              |
| `PI_REASONING_LEVEL` | фактический уровень reasoning: от `off` до `max`                |

Например, tool или скрипт может вернуть проверяемый контекст без догадок по system prompt:

```bash
printf '%s/%s\n' "$PI_PROVIDER" "$PI_MODEL"
printf 'reasoning=%s session=%s\n' "$PI_REASONING_LEVEL" "$PI_SESSION_ID"
```

Это именно окружение LLM-callable bash tool: пользовательские `!` и `!!` его не получают. Кастомный `createBashTool()` получает его по умолчанию до `spawnHook`; при необходимости изоляции опция `exposeSessionEnvironment: false` отключает инъекцию и удаляет унаследованные `PI_SESSION_*`/model-переменные, чтобы вложенный Pi не увидел stale metadata.

Для прямой RPC-команды `bash` добавлен event `bash_execution_update`. В запросе надо указать `id`, и каждое streamed output chunk возвращается с тем же идентификатором — это решает сопоставление вывода при параллельных командах:

```json
{ "id": "build-42", "type": "bash", "command": "pnpm test" }
```

```json
{
  "type": "bash_execution_update",
  "id": "build-42",
  "delta": "PASS src/example.test.ts\n"
}
```

Итоговый `response` по-прежнему содержит final output, exit code, cancellation и truncation, а потоковые события приходят даже когда итоговый output обрезан.

## Несколько исправлений, важных для долгих сессий и провайдеров

- DNS-ошибки `getaddrinfo`, `ENOTFOUND` и `EAI_AGAIN` теперь классифицируются как retryable, в том числе в обёрнутом Bedrock error. Agent session проходит через уже существующий automatic retry path ([#6946](https://github.com/earendil-works/pi/pull/6946)).
- Для OpenRouter Anthropic-style cache markers теперь допустимы на завершающих OpenAI `tool` messages, а cache prefix продвигается через turns, содержащие только tool results ([#6941](https://github.com/earendil-works/pi/pull/6941)).
- При `previous_response_not_found` в OpenAI Codex WebSocket Pi очищает cached `previous_response` и повторно открывает connection вместо того, чтобы продолжать отсутствующую цепочку ([#6955](https://github.com/earendil-works/pi/pull/6955)).
- Compaction и branch summaries используют свежие routing session IDs, отключают prompt-cache writes там, где это поддержано, и следуют общей retry policy на transient failures ([#6618](https://github.com/earendil-works/pi/pull/6618)).
- Каталоги reasoning levels теперь отражают только effort values, которые провайдер подтвердил в models.dev, а не предполагаемый стандартный набор ([#6928](https://github.com/earendil-works/pi/pull/6928)).

Для авторов agent-интеграций 0.82.0 даёт два практических инструмента: можно требовать от provider валидные tool arguments там, где цена неправильного вызова высока, и при этом не выдавать поддержку строгого/grammar режима за универсальную. Для пользователей CLI менее заметные изменения тоже конкретны: OpenRouter и Kimi Code можно подключить через штатный login flow, а внешние bash/RPC-клиенты наконец получают идентифицируемый контекст и вывод в процессе выполнения.
