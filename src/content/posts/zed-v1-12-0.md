Zed выпустил minor-релиз [`v1.12.0`](https://github.com/zed-industries/zed/releases/tag/v1.12.0). Это не pointer-only релиз: основной источник — GitHub Release и связанные PR. Самые заметные изменения касаются повседневного рабочего цикла: Git Panel умеет разделять staged и unstaged части, File Finder и Text Finder открывают несколько результатов за раз, а `format_on_save` можно ограничить строками, которые отличаются от Git-базы.

Для разбора проверены release notes и реализации PR [`#59884`](https://github.com/zed-industries/zed/pull/59884), [`#59931`](https://github.com/zed-industries/zed/pull/59931), [`#60919`](https://github.com/zed-industries/zed/pull/60919), [`#49964`](https://github.com/zed-industries/zed/pull/49964), [`#59838`](https://github.com/zed-industries/zed/pull/59838), [`#60339`](https://github.com/zed-industries/zed/pull/60339), [`#60749`](https://github.com/zed-industries/zed/pull/60749), [`#57207`](https://github.com/zed-industries/zed/pull/57207) и [`#58225`](https://github.com/zed-industries/zed/pull/58225).

## Git Panel: отдельные секции `Staged` и `Unstaged`

PR [`#59884`](https://github.com/zed-industries/zed/pull/59884) добавляет третий вариант `git_panel.group_by`: `staging`. Раньше панель умела не группировать файлы вовсе или группировать их по статусам; теперь в View Options можно выбрать **Staging** и получить секции `Staged` и `Unstaged`.

```json
{
  "git_panel": {
    "group_by": "staging"
  }
}
```

Это не только другая сортировка списка. В коде добавлены отдельные `StagedDiff` и `UnstagedDiff` как цели открытия. Если у файла есть и проиндексированные, и рабочие изменения, он показывается в обеих секциях: клик по строке в `Staged` открывает индексный diff, а в `Unstaged` — diff рабочего дерева. Тесты PR отдельно проверяют этот случай с частично staged файлом.

У секций и файлов появились направленные действия: кнопка в `Staged` снимает с индекса, а в `Unstaged` добавляет в индекс. Поэтому действие над заголовком секции применяется сразу к её набору файлов, а не угадывает состояние по смешанному списку. Конфликты остаются отдельной секцией.

В том же Git-блоке релиза добавлены **Restore** и **Restore All** для unstaged diff ([`#60639`](https://github.com/zed-industries/zed/pull/60639)), теги в tooltip blame ([`#60757`](https://github.com/zed-industries/zed/pull/60757)) и контекстное меню Git Graph в History tab ([`#60713`](https://github.com/zed-industries/zed/pull/60713)).

## File Finder и Text Finder: выбрать несколько результатов и открыть одной командой

PR [`#59931`](https://github.com/zed-industries/zed/pull/59931) переводит picker с единственного выбранного индекса на набор `selected_indices` и добавляет delegate-метод `confirm_multi`. Обычный клик сохраняет прежнее поведение — открывает файл сразу. Multi-select включается без открытия результата:

- `Cmd`-click в macOS или `Ctrl`-click в Linux/Windows добавляет строку в выбор;
- `Tab` добавляет текущий результат и переводит фокус к следующему;
- `Enter` открывает все выбранные элементы;
- выбор очищается при закрытии picker'а или изменении поискового запроса.

PR [`#60919`](https://github.com/zed-industries/zed/pull/60919) делает возможность заметнее: в строке поиска появились checkboxes и кнопка multi-select, а shortcut — `Cmd+Shift+S` на macOS и `Ctrl+Shift+S` на Linux/Windows. Release notes подтверждают, что это работает и для File Finder, и для Text Finder.

Это удобно, например, когда после поиска надо открыть несколько однотипных тестов или все совпадения конфигурационного файла, не повторяя поиск для каждого пути.

## `format_on_save`: форматировать только Git-изменения

PR [`#49964`](https://github.com/zed-industries/zed/pull/49964) добавляет два значения `format_on_save`:

```json
{
  "format_on_save": "modifications"
}
```

`"modifications"` форматирует только строки, отличающиеся от Git-базы. Для этого нужен доступный source control diff; если diff нельзя получить, форматирование пропускается. Режим рассчитан на legacy-код, где сохранение маленькой правки не должно превращать весь файл в гигантский formatting diff.

Второй режим более терпим к неполной Git-информации:

```json
{
  "format_on_save": "modifications_if_available"
}
```

Он также пытается применить range formatting к изменённым строкам, но возвращается к форматированию всего файла для untracked файлов, отсутствующего source control или LSP без range-formatting. Zed также импортирует эквивалентную VS Code-настройку `editor.formatOnSaveMode`.

Обе настройки согласованы с post-save очисткой: `remove_trailing_whitespace_on_save` и `ensure_final_newline_on_save` в modification-режимах затрагивают только изменённые строки. Это предотвращает лишний шум в diff, например в старом файле с историческими trailing spaces.

## Результаты LSP можно показывать в picker с preview

PR [`#59838`](https://github.com/zed-industries/zed/pull/59838) добавляет настройку `lsp_results_location` для `editor: find all references`, `editor: go to definition` и `editor: go to implementation`. По умолчанию остаётся прежний `multi_buffer`, но можно сделать picker глобальным:

```json
{
  "lsp_results_location": "picker"
}
```

Или переопределить место выдачи в конкретном keybinding через параметр `open_results_in`:

```json
[
  {
    "bindings": {
      "ctrl-alt-d": [
        "editor: go to definition",
        { "open_results_in": "picker" }
      ]
    }
  }
]
```

Picker группирует результаты по файлу, показывает номер строки и подсвеченную строку исходника; текст и путь можно фильтровать, preview обновляется при смене selection. `Enter` открывает результат, `Cmd/Ctrl+Enter` — в split. Одиночный результат, как и раньше, открывается сразу, а пустой запрос показывает toast. Если definition не найдена, сохранён fallback к поиску references, когда он настроен.

## Фильтры в CSV preview работают в фоне и комбинируются между столбцами

PR [`#60339`](https://github.com/zed-industries/zed/pull/60339) добавляет кнопки фильтра в заголовки столбцов CSV preview. Меню показывает уникальные значения и количество строк; значения можно сортировать `A-Z, then Count` либо `Count, then A-Z`.

Фильтры составляются по столбцам: выбранные значения одного столбца ограничивают строки, а фильтр другого столбца дополнительно пересекает этот набор. Пункты, которые уже не могут дать результатов из-за ранее включённого фильтра, помечаются как `Hidden by other filters`; для столбца есть `Clear all`.

Важная деталь реализации — пересчёт отображаемых строк уходит в background task. Новое действие отменяет прежний незавершённый расчёт, затем обновляет display-to-data mapping и список. Поэтому фильтрация больших CSV не должна блокировать UI thread, а scrollbar сразу получает приблизительную равномерную высоту строк вместо полного синхронного измерения таблицы.

## Custom Anthropic-модели: adaptive thinking и fast mode

В AI-разделе релиза две настройки адресованы тем, кто описывает Anthropic-compatible модели вручную в `language_models.anthropic.available_models`.

PR [`#57207`](https://github.com/zed-industries/zed/pull/57207) добавляет `adaptive` в `mode.type`. До этого custom-модель можно было только включить или выключить для extended thinking; ветка AdaptiveThinking в provider существовала, но была недостижима из settings. Теперь она доступна конфигурацией:

```json
{
  "language_models": {
    "anthropic": {
      "available_models": [
        {
          "name": "claude-opus-4-7",
          "display_name": "Claude Opus 4.7",
          "max_tokens": 1000000,
          "max_output_tokens": 128000,
          "mode": { "type": "adaptive" }
        }
      ]
    }
  }
}
```

Такой вариант особенно нужен proxy-провайдерам, чей `/v1/models` не возвращает блок `capabilities`: Zed сможет показать adaptive-thinking UI с выбором effort `low`, `medium`, `high`, `xhigh` и `max` для модели, описанной вручную.

PR [`#58225`](https://github.com/zed-industries/zed/pull/58225) добавляет ещё одно необязательное поле — `supports_fast_mode`. Раньше Zed снимал `speed: "fast"` с любой custom Anthropic-модели, даже если endpoint поддерживал fast mode. При `true` модель получает capability, а клиент автоматически добавляет нужный Anthropic beta header:

```json
{
  "language_models": {
    "anthropic": {
      "available_models": [
        {
          "name": "my-anthropic-proxy-model",
          "supports_fast_mode": true
        }
      ]
    }
  }
}
```

Если поле не задано, встроенное распознавание fast-mode моделей остаётся fallback'ом.

## ACP может запрашивать структурированный ввод без feature flag

PR [`#60749`](https://github.com/zed-industries/zed/pull/60749) снимает feature flag с ACP elicitations. ACP-агенты теперь по умолчанию могут запускать elicitation — запрос к пользователю на структурированный ввод, а не только писать свободный текст в чат. В release notes это сформулировано как возможность «collect structured user input»; это важно для внешних агентов, которым нужно запросить параметр или подтверждение в определённой форме.

## Что проверить после обновления

1. Для review-процесса включите `git_panel.group_by: "staging"`: частично staged файлы теперь можно просматривать в контексте именно index или worktree diff.
2. Если форматтер регулярно переписывает старые файлы целиком, начните с `format_on_save: "modifications"`; для untracked файлов и LSP без range formatting подойдёт `modifications_if_available`.
3. Если multi-location LSP навигация удобнее в компактном окне, поставьте `lsp_results_location: "picker"` или используйте `open_results_in` в отдельном shortcut.
4. При custom Anthropic endpoint проверьте `mode.type: "adaptive"` и `supports_fast_mode`, но включайте fast mode только если backend действительно его поддерживает.
