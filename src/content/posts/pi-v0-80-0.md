[`pi`](https://github.com/earendil-works/pi) выпустил минорный релиз [`v0.80.0`](https://github.com/earendil-works/pi/releases/tag/v0.80.0). Главная тема версии — миграция `@earendil-works/pi-ai` от старого глобального API к явной модели `Models`: провайдеры теперь владеют каталогом моделей, auth и stream-реализацией, а прежние `stream`/`complete`/`getModel`/`registerApiProvider` уехали в временный compat-entrypoint.

Источник: GitHub Release [`earendil-works/pi@v0.80.0`](https://github.com/earendil-works/pi/releases/tag/v0.80.0), compare [`v0.79.10...v0.80.0`](https://github.com/earendil-works/pi/compare/v0.79.10...v0.80.0), changelog в `packages/ai/CHANGELOG.md` и `packages/agent/CHANGELOG.md`, а также связанные PR/issue: [#5999](https://github.com/earendil-works/pi/pull/5999), [#5784](https://github.com/earendil-works/pi/pull/5784), [#5526](https://github.com/earendil-works/pi/pull/5526), [#5973](https://github.com/earendil-works/pi/issues/5973), [#5965](https://github.com/earendil-works/pi/issues/5965), [#5967](https://github.com/earendil-works/pi/issues/5967) и [#5953](https://github.com/earendil-works/pi/issues/5953).

## Старый root API `pi-ai` переехал в `/compat`

Самое важное breaking-изменение для TypeScript-кода: root entrypoint `@earendil-works/pi-ai` больше не экспортирует старый глобальный API. В `packages/ai/src/index.ts` остались core-типы, auth-helpers, `createModels()`, `Provider`, `Models`, `ImagesModels`, faux-provider и API-типы. Комментарий в файле прямо фиксирует новый контракт: root side-effect-free, без built-in provider factories, OAuth implementations, `api-registry` и compat.

Прежняя поверхность вынесена в `packages/ai/src/compat.ts`. Там временно живут:

- `stream()`, `complete()`, `streamSimple()`, `completeSimple()`;
- `getModel()`, `getModels()`, `getProviders()` как deprecated aliases к built-in catalog reads;
- `registerApiProvider()`, `getApiProvider()`, `unregisterApiProviders()`;
- `getEnvApiKey()`, image API и lazy wrappers для конкретных transport API.

Минимальная миграция для кода, который пока не готов к новой архитектуре:

```ts
// Было до 0.80.0
import { completeSimple, getModel } from "@earendil-works/pi-ai";

// В 0.80.0 как временный путь совместимости
import { completeSimple, getModel } from "@earendil-works/pi-ai/compat";
```

Для расширений runtime-поведение специально не ломают: loader в `packages/coding-agent/src/core/extensions/loader.ts` резолвит `@earendil-works/pi-ai` в `_bundledPiAiCompat` и в Bun binary virtual modules, и в Node/Jiti aliases. Поэтому уже собранные расширения продолжают работать. Но если исходники расширения typecheck'ятся против опубликованных типов `pi-ai`, импорты старого API нужно поменять на `@earendil-works/pi-ai/compat` или сразу перейти на новый `Models` API. Релиз подчёркивает, что compat-entrypoint и alias в loader будут удалены позже.

## Новый путь: `Models` collection и provider factories

Новый runtime-контракт описан в `packages/ai/src/models.ts`: `Provider` теперь является конкретной единицей исполнения. Он содержит `id`, `name`, `baseUrl`, `auth`, синхронный `getModels()`, опциональный `refreshModels()` для динамических провайдеров и методы `stream()`/`streamSimple()`. `Models` — это коллекция провайдеров, которая умеет искать модели, резолвить auth и делегировать запрос владельцу модели.

Практический quick start из README после релиза выглядит так:

```ts
import { Type, type Context, type Tool } from "@earendil-works/pi-ai";
import { builtinModels } from "@earendil-works/pi-ai/providers/all";

const models = builtinModels();
const model = models.getModel("openai", "gpt-4o-mini")!;

const tools: Tool[] = [
  {
    name: "get_time",
    description: "Get the current time",
    parameters: Type.Object({ timezone: Type.Optional(Type.String()) }),
  },
];

const context: Context = {
  systemPrompt: "You are a helpful assistant.",
  messages: [
    { role: "user", content: "What time is it?", timestamp: Date.now() },
  ],
  tools,
};

const response = await models.complete(model, context);
```

Если приложению не нужны все built-ins, можно импортировать только конкретные factories. В `packages/ai/src/providers/all.ts` видно новый список factories: `openaiProvider()`, `anthropicProvider()`, `openrouterProvider()`, `githubCopilotProvider()`, `amazonBedrockProvider()`, `opencodeGoProvider()`, `zaiProvider()` и другие. Это важно для bundling: root package больше не подтягивает все каталоги и SDK-обвязки только из-за одного импорта.

Для тестов появился provider-oriented faux API. `fauxProvider()` возвращает handle с `provider`, `models`, `getModel()`, `setResponses()` и счётчиком вызовов; его можно положить в коллекцию через `models.setProvider(faux.provider)` вместо регистрации глобального mock API.

## AgentHarness теперь принимает `models`, а не per-request key resolver

В `packages/agent/CHANGELOG.md` релиз помечен как breaking для agent-core. `AgentHarnessOptions.models` стал обязательным и единственным auth path. Раньше harness мог получать ключи через `getApiKeyAndHeaders`; в 0.80.0 этот callback удалён, а streaming, compaction и branch summarization идут через переданный `Models` instance.

Это видно в `packages/agent/src/harness/agent-harness.ts`: класс хранит `readonly models: Models`, а `createStreamFn()` вызывает `this.models.streamSimple(model, context, ...)`. Отдельное склеивание headers из `getApiKeyAndHeaders` исчезло — auth должен быть выражен на уровне provider auth (`ApiKeyAuth`/`OAuthAuth`) в коллекции.

Схематичная миграция для harness-кода:

```ts
import { createModels } from "@earendil-works/pi-ai";
import { openaiProvider } from "@earendil-works/pi-ai/providers/openai";
import { AgentHarness } from "@earendil-works/pi-agent-core";

const models = createModels();
models.setProvider(openaiProvider());

const harness = new AgentHarness({
  env,
  session,
  models,
  // getApiKeyAndHeaders больше не передаётся
});
```

Также API compaction поменял сигнатуры: `compact()`, `generateSummary()` и `generateBranchSummary()` принимают `Models`, а не явные `apiKey`/`headers`. `StreamFn` теперь структурно совместим с `Models.streamSimple`.

## Провайдеры и auth стали строже и понятнее

В релиз вошло несколько изменений, которые выглядят как мелкие fixes, но все они связаны с новой моделью провайдеров.

Во-первых, Cloudflare providers теперь требуют account/gateway configuration и прогоняют built-in compat calls через provider auth. Это закрывает ситуацию, когда compat-вызов мог обойти те же правила auth, что и новый `Models` path.

Во-вторых, provider-scoped environment values доходят до inherited `Models`/`ImagesModels` calls и compat API-key injection. Для кода с несколькими провайдерами это важно: env overrides должны применяться в scoped context, а не теряться при переходе через compat-слой.

В-третьих, Amazon Bedrock endpoint resolution теперь учитывает scoped `AWS_PROFILE`. Если пользователь выбирает профиль в окружении конкретного provider context, Bedrock endpoint строится из него, а не из глобального default.

Наконец, issue [#5953](https://github.com/earendil-works/pi/issues/5953) закрывает раздражающий кейс с `models.json`: кастомный provider больше не обязан иметь дублирующий `apiKey`, если валидные credentials уже лежат в `~/.pi/agent/auth.json`. Пример из issue раньше падал при загрузке:

```json
{
  "providers": {
    "ollama-cloud": {
      "api": "openai-completions",
      "baseUrl": "https://ollama.com/v1",
      "models": [{ "id": "glm-5.2" }]
    }
  }
}
```

До фикса приходилось добавлять placeholder вроде `"apiKey": "from-auth-json"`. После 0.80.0 stored auth может удовлетворить проверку без такого мусорного поля.

## `Ctrl+J` для новой строки и яснее название ZAI

Интерактивный UX тоже получил пару точечных правок.

Первое изменение — `Ctrl+J` добавлен как default keybinding для новой строки рядом с `Shift+Enter`. Это полезно в терминалах, где `Shift+Enter` не всегда стабильно проходит через emulator, tmux или удалённую сессию. Теперь у пользователя есть второй стандартный способ вставить newline в prompt без отправки сообщения.

Второе изменение пришло из issue [#5965](https://github.com/earendil-works/pi/issues/5965): displayed label провайдера `zai` переименован в `ZAI Coding Plan (Global)`. Причина не косметическая: base URL у built-in `zai` — `https://api.z.ai/api/coding/paas/v4`, то есть global Coding Plan endpoint, а рядом существует отдельный China provider. Новая подпись снижает риск выбрать не тот Z.AI endpoint.

## Threaded sessions сортируются по активности всего поддерева

PR [#5784](https://github.com/earendil-works/pi/pull/5784) меняет логику `buildSessionTree()` в `packages/coding-agent/src/modes/interactive/components/session-selector.ts`. Раньше roots и children сортировались только по `session.modified` самого узла. Если у старой parent-сессии были свежие forks, весь subtree всё равно мог оказаться ниже новых одноразовых сессий.

В 0.80.0 у `SessionTreeNode` появилось поле `latestActivity`. Функция `updateLatestActivity()` рекурсивно берёт максимум между `node.session.modified` и всеми дочерними `modified`, а сортировка идёт по `b.latestActivity - a.latestActivity`.

Тест в `packages/coding-agent/test/session-selector-path-delete.test.ts` фиксирует сценарий: `Parent two` старше `Parent one`, но у него есть `Child two` с более свежей датой. В threaded selector весь subtree `Parent two → Child two` теперь отображается выше `Parent one`. Для пользователя это означает меньше скролла при возврате к активной ветке работы.

## Имена сессий больше не сохраняют переносы строк

PR [#5999](https://github.com/earendil-works/pi/pull/5999) закрывает баг с session labels. В `packages/agent/src/harness/session/session.ts` и `packages/coding-agent/src/core/session-manager.ts` добавлена одинаковая нормализация:

```ts
const sanitizedName = name.replace(/[\r\n]+/g, " ").trim();
```

То есть строка `" hello\nworld\r\nagain "` сохраняется как `"hello world again"`. В interactive mode после `setSessionName()` UI перечитывает нормализованное имя из session manager; если оно отличается от пользовательского ввода, показывается warning `Session name was normalized ...`, а строка `Session name set:` использует уже очищенное имя. Отдельный regression test проверяет и прямой `AgentSession.setSessionName`, и вызов из extension API `pi.setSessionName()`.

## OpenAI Responses: terminal events, reconnect и usage fixes

В релизе несколько исправлений вокруг inherited OpenAI Responses/Codex transports.

PR [#5526](https://github.com/earendil-works/pi/pull/5526) меняет критерий успешного stream completion: OpenAI Responses stream должен закончиться terminal response event. `response.incomplete` трактуется как length stop, `response.failed` — как provider error. Это защищает от ситуации, когда stream случайно оборвался, но Pi уже посчитал turn успешным и пользователю приходилось вручную писать `continue`.

Тот же PR исправляет context accounting после malformed/partial responses: all-zero assistant usage теперь игнорируется при оценке coding-agent context и compaction. Иначе обрезанный ответ мог испортить счётчик контекста.

Issue [#5973](https://github.com/earendil-works/pi/issues/5973) закрывает отдельный WebSocket case для OpenAI Codex Responses. Если сессия упирается в лимит OpenAI `websocket_connection_limit_reached` до начала assistant output, Pi теперь один раз открывает новый WebSocket и повторяет turn с полным контекстом. Если reconnect не удаётся, остаётся прежний fallback-path.

## GLM-5.2 в OpenCode Go получил `xhigh`

Issue [#5967](https://github.com/earendil-works/pi/issues/5967) добавляет max reasoning effort для `opencode-go/glm-5.2`. В сгенерированном `packages/ai/src/providers/opencode-go.models.ts` у модели теперь есть `reasoning: true` и карта уровней:

```ts
thinkingLevelMap: {
  off: null,
  minimal: null,
  low: null,
  medium: null,
  high: "high",
  xhigh: "max",
}
```

Практический эффект: выбор `--thinking xhigh` для GLM-5.2 на OpenCode Go отправляет провайдеру max reasoning effort, а не ограничивается обычным high.

## Что ещё важно при обновлении

- `pi --resume` теперь загружает user package themes и корректно резолвит automatic light/dark theme settings. Это исправляет сценарий, где возобновлённая сессия не видела тему пользователя.
- При extension-related crash/startup failure сообщение теперь предлагает перезапуск с `pi -ne`, то есть с выключенными extensions. Это даёт пользователю конкретный recovery-command вместо абстрактной ошибки.
- Selective-provider entrypoints `@earendil-works/pi-ai/base` и `@earendil-works/pi-agent-core/base` удалены. Новый путь — root packages плюс явные `Models` provider factories.

Для прикладных пользователей `pi` 0.80.0 выглядит как набор UX-фиксов, но для авторов интеграций это миграционный релиз: стоит уже сейчас заменить root imports старого `pi-ai` API на `/compat` или перейти на `createModels()` и factories, потому что compat и loader alias объявлены временными.
