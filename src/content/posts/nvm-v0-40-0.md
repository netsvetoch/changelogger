[`nvm` 0.40.0](https://github.com/nvm-sh/nvm/releases/tag/v0.40.0) — минорный релиз Node Version Manager, который в основном улучшает повседневный CLI-сценарий: зафиксировать Node-версию проекта, читать более живые `.nvmrc`-файлы и ходить в приватные зеркала дистрибутивов Node.

Источник: GitHub Release [`nvm-sh/nvm@v0.40.0`](https://github.com/nvm-sh/nvm/releases/tag/v0.40.0). Для конкретики я проверил compare [`v0.39.7...v0.40.0`](https://github.com/nvm-sh/nvm/compare/v0.39.7...v0.40.0), PR [#2869](https://github.com/nvm-sh/nvm/pull/2869), [#3380](https://github.com/nvm-sh/nvm/pull/3380), [#3394](https://github.com/nvm-sh/nvm/pull/3394), а также diff `nvm.sh` и `README.md` в релизном теге.

## `nvm install --save` и `nvm use --save` сами пишут `.nvmrc`

Главное пользовательское изменение из [#2869](https://github.com/nvm-sh/nvm/pull/2869) — у `nvm install` и `nvm use` появился флаг `--save`. В help-тексте он описан явно: для `install` — «After installing, write the specified version to .nvmrc», для `use` — «Writes the specified version to .nvmrc». Короткий алиас тоже есть: `-w`.

```sh
nvm install --save 20
# или короче
nvm use -w 20
```

Внутри `nvm.sh` появился helper `nvm_write_nvmrc()`: он сначала прогоняет переданную строку через `nvm_version`, отбрасывает неразрешимые значения вроде `N/A`, затем пишет нормализованную версию в `$PWD/.nvmrc`. Поэтому при успешном сценарии в файл попадает не произвольный ввод, а версия, которую сам nvm считает валидной.

Практический эффект: после выбора Node для проекта больше не нужно отдельно делать `node -v > .nvmrc` или вручную редактировать файл. Это особенно удобно в bootstrap-скриптах репозитория:

```sh
nvm install --lts --save
# .nvmrc будет обновлён выбранной LTS-версией
```

Тесты релиза покрывают несколько деталей поведения: `nvm install --save` создаёт валидный `.nvmrc`, `nvm use --save` работает, когда `.nvmrc` есть в родительской директории, `--silent --save` не печатает лишний stdout, а `nvm_write_nvmrc` падает на невалидной версии и на недоступном для записи файле.

## `.nvmrc` теперь можно комментировать

До 0.40.0 README требовал, чтобы `.nvmrc` содержал только версию и завершающий перевод строки. В релизе это правило стало практичнее: `.nvmrc` может содержать комментарии, пустые строки и пробелы вокруг значимых строк. Комментарий начинается с `#`; сам разделитель и всё после него игнорируются при парсинге.

```text
# Версия Node для локальной разработки и CI
20.11.1

# служебные строки key=value пока игнорируются nvm
owner = frontend-platform
```

В `nvm.sh` это реализовано новой функцией `nvm_process_nvmrc()`: она удаляет `#...`, trim'ит пробелы, выбрасывает пустые строки и требует, чтобы после такой обработки осталась ровно одна «bare» версия. Key/value-строки с `=` разрешены и игнорируются, но README прямо помечает их как зарезервированные для будущего поведения.

Есть важное ограничение: alias больше не может содержать `#`. В diff `nvm.sh` добавлена проверка, которая возвращает ошибку `Aliases with a comment delimiter (#) are not supported.` Это цена за то, что `#` теперь стал синтаксисом комментариев в `.nvmrc`.

## `NVM_AUTH_HEADER` для приватных зеркал Node

[#3380](https://github.com/nvm-sh/nvm/pull/3380) добавляет поддержку переменной окружения `NVM_AUTH_HEADER`. Она нужна, когда `NVM_NODEJS_ORG_MIRROR` или другое зеркало требует авторизацию при скачивании Node-дистрибутивов.

Документация в README теперь показывает такой сценарий:

```sh
NVM_AUTH_HEADER="Bearer secret-token" nvm install node
```

В `nvm_download()` поведение различается по backend'у:

- для `curl` nvm строит `--header "Authorization: ..."`; перед этим значение проходит через новый `nvm_sanitize_auth_header()`, который оставляет только буквы, цифры, `:`, `;`, `_`, `.`, пробел и `-`;
- для `wget` к аргументам добавляется `--header "${NVM_AUTH_HEADER}"`.

Это изменение полезно для корпоративных сетей, где прямой доступ к `nodejs.org/dist` закрыт, а внутреннее зеркало защищено bearer-токеном или другим Authorization header.

## Меньше поломок вокруг npm, zsh и default-packages

В релиз вошло несколько небольших, но конкретных исправлений для сценариев, где nvm раньше мог вести себя неожиданно.

`install-latest-npm` теперь ставит `npm@8.6` для старых диапазонов Node, где раньше выбирался `npm@8`. В комментарии к коду причина сформулирована прямо: `^8.7 breaks npm ls on file: deps`. То есть для Node v12, v14.13–v14.16 и v16.0–v16.12 nvm больше не будет случайно подтягивать npm 8.7+ с регрессией в `npm ls` для `file:`-зависимостей.

`nvm_get_default_packages` переписан на `awk`. Старый код вручную создавал временную директорию, добавлял завершающую новую строку через `sed`, читал файл построчно и собирал список пакетов. Новый вариант одной `awk`-программой пропускает пустые строки и строки-комментарии, ругается на строки с пробелами и печатает пакеты через пробел. Для пользователя контракт тот же — один пакет на строку в `$NVM_DIR/default-packages`, — но обработка стала короче и предсказуемее.

[#3394](https://github.com/nvm-sh/nvm/pull/3394) исправляет конфликт с zsh global aliases в help-тексте. Пример из PR:

```sh
alias -g C='| clipcopy'
```

Раньше такие короткие алиасы могли раскрыться внутри вызовов цветного help-текста. В diff аргументы `nvm_wrap_with_color_code` теперь заключены в кавычки (`'r'`, `'y'`, `'C'` и т.д.), чтобы zsh не пытался интерпретировать их как global alias.

## Цвета и uninstall-документация тоже стали точнее

Рефакторинг `nvm_has_colors` теперь учитывает `NVM_NO_COLORS=--no-colors` прямо в проверке цветов, а не размазывает это условие по отдельным местам вывода. Поэтому функции печати версий и алиасов могут полагаться на один predicate: если пользователь отключил цвета, `nvm_has_colors` вернёт false даже при терминале с `tput colors >= 8`.

Документация uninstall-сценария также стала безопаснее. Вместо прямого удаления текущего `$NVM_DIR` README теперь сначала сохраняет путь, выгружает shell-функции через `nvm unload`, а затем удаляет сохранённую директорию:

```sh
nvm_dir="${NVM_DIR:-~/.nvm}"
nvm unload
rm -rf "$nvm_dir"
```

Это важно, потому что `nvm unload` сам очищает `NVM_DIR`; без предварительного сохранения переменной инструкция могла потерять путь к установленному nvm.

## Кому стоит обновиться

0.40.0 стоит поставить командам, которые хранят Node-версию в репозитории: `--save` убирает ручное редактирование `.nvmrc`, а поддержка комментариев позволяет объяснять, почему выбрана конкретная версия. Пользователям приватных зеркал релиз даёт официальный путь для Authorization header через `NVM_AUTH_HEADER`. Если вы поддерживаете старые Node-линейки, обновление также снижает риск нарваться на проблемный `npm` 8.7+ при `install-latest-npm`.
