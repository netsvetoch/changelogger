Вышел минорный релиз [Knip 6.26.0](https://github.com/webpro-nl/knip/releases/tag/knip%406.26.0) — инструмента, который находит неиспользуемые файлы, зависимости и exports в JavaScript- и TypeScript-проектах. Главная тема обновления — поддержка инструментов вокруг Vue и Vite: Knip теперь лучше понимает виртуальные модули, auto-imports, файловые маршруты, layouts, PWA-конфигурацию и команды из конфигов.

## Vue auto-imports теперь видны анализатору

В проектах на Vue часть зависимостей не появляется в исходном коде как обычный `import`. Плагины `unplugin-auto-import` и `unplugin-vue-components` генерируют `.d.ts`-файлы, а компоненты используются прямо в шаблоне `.vue`. Раньше статический анализ мог не связать такой вызов или тег с реальным исходным файлом.

В 6.26.0 появилась общая machinery в `packages/knip/src/plugins/_vue/auto-import.ts`:

- `unplugin-auto-import` читает `auto-imports.d.ts` и синтетически добавляет импорты composables в `.ts` и `.vue`;
- `unplugin-vue-components` читает `components.d.ts` и связывает используемые в шаблоне теги с компонентами;
- `unplugin-vue-markdown` применяет похожий compiler к `.md`, отслеживая компоненты в Markdown, но не трактуя fenced code blocks как реальные usages;
- `unplugin-vue-router` считает `src/pages/**/*.{vue,md}` production entries, потому что маршруты подхватываются файловой структурой, а не импортируются вручную;
- для `vue-router` поддержан случай с generated TypeScript-файлом, содержащим маркер `Generated by vue-router`.

Например, следующий код теперь может быть проанализирован без ручного добавления `useCounter` в `knip.json`, если рядом есть сгенерированный `auto-imports.d.ts`:

```vue
<script setup lang="ts">
const counter = useCounter();
</script>

<template>
  <div>{{ counter }}</div>
</template>
```

А компонент, подключённый через `components.d.ts`, не нужно импортировать явно:

```vue
<template>
  <AppButton label="Сохранить" />
</template>
```

Важна и граница этой автоматизации: plugin не превращает весь каталог в entry. Например, в тестах Knip отдельно проверяет, что неиспользуемый composable или компонент по-прежнему может попасть в отчёт.

## Новые плагины для Vite-экосистемы

### `vite-plugin-pages` и layouts

Плагин `vite-plugin-pages` добавляет production entries для `src/pages/**/*.{vue,jsx,tsx,md,mdx}` и игнорирует виртуальные импорты `~pages`, `~react-pages` и `~solid-pages`. Если в вызове плагина передан собственный `dirs`, Knip извлекает его из AST `vite.config.*`; иначе используется `src/pages`.

Поддержаны также `vite-plugin-vue-layouts-next`, `vite-plugin-vue-layouts` и `vite-plugin-vue-meta-layouts`. Для них аналогично читается `layoutsDirs`, а fallback — `src/layouts/**/*.vue`. Это позволяет не добавлять страницы и layouts в `entry` вручную только потому, что они подключаются виртуальным модулем.

### PWA и i18n

`vite-plugin-pwa` теперь распознаёт `injectManifest` и регистрирует service worker как production entry. Параметры берутся из `vite.config.*`:

```ts
VitePWA({
  strategies: "injectManifest",
  srcDir: "src",
  filename: "sw.ts",
});
```

Если параметры не заданы, используются `public/sw.js`. Поддержан и Nuxt-вариант: для `@vite-pwa/nuxt` Knip читает тот же набор полей из блока `pwa` в `nuxt.config.*`:

```ts
export default defineNuxtConfig({
  pwa: {
    strategies: "injectManifest",
    srcDir: "service-worker",
    filename: "sw.ts",
  },
});
```

Добавлены отдельные плагины для `@vite-pwa/assets-generator` и `@nuxtjs/i18n`. Первый распознаёт `pwa-assets.config.*`, а второй — `i18n.config.*` в корне или конфигурацию внутри каталога `i18n/`. Для `@intlify/unplugin-vue-i18n` виртуальный модуль `@intlify/unplugin-vue-i18n/messages` больше не должен приводить к ложному `unlisted dependency`.

### `vite-plus`

Knip научился читать из `vite.config.*` конфигурацию `vite-plus`:

- `run.tasks[].command` — команды task runner;
- `staged` — команды pre-commit hooks;
- `lint.jsPlugins` — ESLint-compatible plugins из встроенного oxlint.

Команды разбираются AST-парсером и не выполняются. Найденные в них файлы становятся entries, а бинарники — зависимостями; режим `knownBinsOnly` оставляет системные команды вроде `git` вне отчёта. `jsPlugins` ищутся как в отдельном `.oxlintrc.json`/`oxlint.config.ts`, так и в `lint`-блоке `vite.config.ts`.

## `electron-vite` и `esbuild`

Новый `electron-vite` plugin читает `electron.vite.config.*` и анализирует секции `main`, `preload` и `renderer`. Для каждой секции поддерживаются:

- `build.rollupOptions.input`;
- `build.lib.entry`;
- значения-строки, массивы и объекты entries;
- fallback-пути electron-vite: `src/main/index`, `src/preload/index` и `src/renderer/index.html`.

Пример с явными entrypoints:

```ts
export default defineConfig({
  preload: {
    build: {
      rollupOptions: {
        input: { chat: "src/preload/chat.ts" },
      },
    },
  },
});
```

Для `esbuild` plugin анализирует `entryPoints` в `esbuild.config.*` и `esbuild.*`. Скрипт намеренно не выполняется: значения извлекаются из AST, чтобы запуск Knip не запускал сборку. Например, оба файла из такой конфигурации будут зарегистрированы как production entries:

```js
await esbuild.build({
  entryPoints: ["src/extension.ts", "src/worker.ts"],
  bundle: true,
  outdir: "dist",
});
```

## Ещё несколько новых интеграций

В список plugins добавлены:

- `Tauri` — читает `src-tauri/tauri.conf.{json,json5}`, платформенные конфиги и `Tauri.toml`; команды `beforeDevCommand`, `beforeBuildCommand` и `beforeBundleCommand` разбираются как scripts, включая вариант с объектом `{ script, cwd }`;
- `Laravel` — распознаёт Laravel-проект и его frontend entrypoints;
- `Quasar` — учитывает конфигурацию Quasar и связанные entrypoints;
- `oclif` получил поддержку `@oclif/core` и регистрирует `{,src/}commands/**/*.{js,mjs,cjs,jsx,ts,tsx,mts,cts}` как production entries;
- Vitest теперь распознаёт benchmark-файлы как entries;
- Storybook извлекает Babel plugins из AST конфигурации, включая `viteFinal`, где они могут быть переданы в `@vitejs/plugin-react` или `@rolldown/plugin-babel`.

Для oclif это исправляет конкретный сценарий автозагрузки команд: файл `src/commands/db/migrate.ts` может не иметь статического импортёра, потому что его находит runtime oclif. В новой конфигурации такой файл не будет ошибочно отмечен как unused.

## Меньше ложных срабатываний в module graph

Несколько изменений затрагивают не новые интеграции, а точность базового анализа:

- `.d.ts` без static exports, содержащие `declare global`, `declare module` или side-effect augmentation imports, больше не считаются unused при solution-style tsconfig с пустым `files`/`include`;
- package entrypoints, разрешившиеся в каталогах из `.gitignore`, исключаются до регистрации production entry; при source mapping glob строится от корня workspace, поэтому `dist` и другие игнорируемые каталоги не возвращаются в граф;
- `#`-imports при не собранном `node` condition резолвятся к исходному файлу;
- `@vitejs/plugin-vue` и `unplugin-vue` теперь включают Vue SFC compiler даже если Vue подключён транзитивно;
- compiler extensions копируются для каждого workspace, чтобы расширение одного workspace, например `.md`, не влияло на project patterns другого;
- Nitro route types больше не принимаются за карту Vue auto-imports;
- в pnpm и Yarn optional peerDependencies из `packageExtensions` больше не считаются обязательными зависимостями;
- глобальными бинарниками теперь считаются, среди прочего, `az`, `cmd.exe`, `code`, `code-insiders`, `codesign`, `devtunnel`, `dotnet`, `open`, `powershell`, `python`, `python3` и `gh`.

Последний пункт означает, что такие команды в scripts не требуют фиктивной зависимости только для того, чтобы Knip не сообщил о binary. При этом имя пакета с таким же названием не считается источником системного binary, если он не поставляет соответствующий executable.

## Обновление

Для обновления достаточно установить новую минорную версию:

```bash
pnpm add -D knip@6.26.0
pnpm knip
```

Если проект использует Vue auto-imports, файловые маршруты или PWA, сначала стоит убрать локальные `entry` и `ignore`, которые дублируют теперь встроенные правила, и проверить результат. Для нестандартных директорий используйте реальные параметры plugin в `vite.config.*` — Knip читает `dirs`, `layoutsDirs`, `srcDir`, `filename` и entrypoints из конфигурации, а не только применяет фиксированные glob-паттерны.

Внутреннюю миграцию документации самого Knip на Astro 7 и Sätteri можно не считать пользовательским API-изменением: это изменение репозитория документации, а не требование для проектов, которые запускают Knip.
