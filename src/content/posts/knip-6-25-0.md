Knip выпустил минорный релиз [`6.25.0`](https://github.com/webpro-nl/knip/releases/tag/knip%406.25.0). В этом обновлении много точечных исправлений для экосистемных плагинов, но главное пользовательское изменение — новый opt-in тип проблем `cycles`, который ищет циклические зависимости в графе модулей.

Источники для обзора — GitHub Release [`webpro-nl/knip@knip@6.25.0`](https://github.com/webpro-nl/knip/releases/tag/knip%406.25.0), compare [`knip@6.24.0...knip@6.25.0`](https://github.com/webpro-nl/knip/compare/knip@6.24.0...knip@6.25.0), PR [#1812](https://github.com/webpro-nl/knip/pull/1812), [#1847](https://github.com/webpro-nl/knip/pull/1847), [#1852](https://github.com/webpro-nl/knip/pull/1852), [#1859](https://github.com/webpro-nl/knip/pull/1859), [#1860](https://github.com/webpro-nl/knip/pull/1860), [#1864](https://github.com/webpro-nl/knip/pull/1864), [#1865](https://github.com/webpro-nl/knip/pull/1865), [#1869](https://github.com/webpro-nl/knip/pull/1869) и коммиты [`fc7bb1e`](https://github.com/webpro-nl/knip/commit/fc7bb1ecc194e69f9f493bff5b33a4cc807fd788), [`d231d67`](https://github.com/webpro-nl/knip/commit/d231d67a5e8713c2f48f9d695ae12a9dbfa0260e), [`6f18138`](https://github.com/webpro-nl/knip/commit/6f1813809608cd36e44af22583805e686c51c5bd), [`38e4a4a`](https://github.com/webpro-nl/knip/commit/38e4a4a4a06d902cf55457155f666c3b91e05f28), [`1255369`](https://github.com/webpro-nl/knip/commit/125536921f1845eeec1347091b238bb615f329e0), [`e3f0e5c`](https://github.com/webpro-nl/knip/commit/e3f0e5c69641bb7dfab06585251c98e3b3def049), [`9422d06`](https://github.com/webpro-nl/knip/commit/9422d06adb07fd29a3388ee6b037cda47fe28d1a). Релиз минорный, поэтому `featured: false`.

## Новый opt-in issue type `cycles`

PR [#1812](https://github.com/webpro-nl/knip/pull/1812) добавляет в Knip отдельный тип проблем `cycles` для циклических зависимостей. Это не включается в обычный набор проверок автоматически: релизный PR прямо описывает режим как opt-in. Запускать можно через CLI:

```bash
knip --cycles
knip --include cycles
knip --cycles --reporter cycles
```

или через конфигурацию:

```json
{
  "include": ["cycles"],
  "rules": {
    "cycles": "error"
  }
}
```

В `packages/knip/src/types/issues.ts` появился новый ключ `cycles`, а в схеме конфигурации — поддержка `rules.cycles`. Для более тонкой настройки в `cycles` добавлены параметры `allow` и `dynamicImports`: первый нужен для разрешённых цепочек, второй — для включения/исключения динамических импортов из анализа.

Отдельный reporter `cycles` печатает не просто строку с файлом, а дерево переходов по импортам. В PR показан пример такого вывода:

```txt
Circular dependencies (1)

src/i18n/index.ts:1:28
└── import ./middleware → src/i18n/middleware.ts:3:15
    └── re-export ../core/i18n/handler → src/core/i18n/handler.ts:5:10
        └── import ../../i18n → src/i18n/index.ts:1:28 ↩
```

Под капотом новый `findAllCycles` проходит runtime-граф одним DFS-проходом и добавляет цикл при back-edge. Type-only связи не должны превращаться в runtime cycles: PR описывает анализ именно runtime edges.

## pnpm и Yarn теперь учитывают `packageExtensions`

PR [#1847](https://github.com/webpro-nl/knip/pull/1847) закрывает важный случай для менеджеров пакетов: Knip теперь читает `packageExtensions` в pnpm и Yarn и учитывает `peerDependencies`, которые добавлены этим механизмом.

Для pnpm плагин смотрит обе формы конфигурации:

```json
{
  "pnpm": {
    "packageExtensions": {
      "some-plugin@*": {
        "peerDependencies": {
          "react": "*"
        }
      }
    }
  }
}
```

и верхнеуровневую форму:

```json
{
  "packageExtensions": {
    "some-plugin@*": {
      "peerDependencies": {
        "react": "*"
      }
    }
  }
}
```

В `packages/knip/src/plugins/pnpm/index.ts` Knip проходит по `packageExtensions`, берёт ключи из `peerDependencies` и добавляет их как dependency inputs. В Yarn-плагине аналогичная логика появилась для `.yarnrc.yml`: если расширение объявляет peer `@types/react`, Knip не должен считать этот пакет неиспользуемым только потому, что связь описана менеджером пакетов, а не обычным `import`.

## Yarn `nodeLinker: pnpm`: имя пакета берётся из `package.json`, а не из `.store`

PR [#1852](https://github.com/webpro-nl/knip/pull/1852) исправляет false positive в проектах на Yarn Berry с `nodeLinker: pnpm`. В такой раскладке реальные файлы лежат не в классическом `node_modules/<pkg>/...`, а внутри store:

```txt
node_modules/.store/<flattened-locator>/package/...
```

Раньше `getPackageNameFromFilePath` выводил имя пакета из последнего сегмента после `node_modules`, и для таких путей получался буквально пакет `.store`. Это всплывало, например, когда плагин резолвил конфиг через `dirname(require.resolve("<addon>/package.json"))`.

Исправление не пытается парсить имя из папки store: автор PR отдельно объяснил, что flattened locator неоднозначен для scoped-пакетов и протокольных маркеров. Вместо этого Knip, когда видит сегмент `.store`, читает `package/package.json` внутри store-пакета и берёт поле `name`. В тестовом fixture покрыты оба класса неоднозначности:

```txt
node_modules/.store/@remix-run-router-npm-1.19.2-abc123/package/package.json
node_modules/.store/write-file-atomic-npm-5.0.1-def456/package/package.json
```

Для первого случая корректное имя — `@remix-run/router`, для второго — `write-file-atomic`. Если `package.json` прочитать нельзя, поведение откатывается к прежнему `.store`, а обычный `node_modules` не получает лишних синхронных чтений: результат кэшируется по store root.

## Алиасы в glob-импортах и `require.context`

Сразу несколько изменений уменьшают шум в проектах на Vite, webpack и rspack.

Коммит [`fc7bb1e`](https://github.com/webpro-nl/knip/commit/fc7bb1ecc194e69f9f493bff5b33a4cc807fd788) добавляет поддержку aliases в `import.meta.glob`. Новый helper `resolveImportGlobs` умеет:

- резолвить начало glob-паттерна через `tsconfig.paths` и Vite `resolve.alias`;
- сохранять относительные и negated patterns;
- интерпретировать root-relative паттерны от Vite root;
- работать с wildcard, non-wildcard и multi-target alias mappings.

Пример сценария, который теперь должен попадать в граф анализа:

```ts
const pages = import.meta.glob("@/pages/**/-page.ts");
const publicRoutes = import.meta.glob("/routes/**/*.ts");
const withoutPrivate = import.meta.glob([
  "@routes/**/*.ts",
  "!@routes/admin/**",
]);
```

Коммит [`d231d67`](https://github.com/webpro-nl/knip/commit/d231d67a5e8713c2f48f9d695ae12a9dbfa0260e) переносит похожую идею на `require.context`: Knip собирает directory glob во время static analysis, а на этапе build резолвит его через те же aliases. Важно, что visitor теперь регистрируется и в Rspack-плагине, не только в webpack. Поэтому entry-файлы, найденные через `require.context("@forms", true, /\\.ts$/)`, больше не теряются только из-за alias.

## Next.js: `src/app` и `src/pages` больше не помечаются entry, если Next их игнорирует

PR [#1859](https://github.com/webpro-nl/knip/pull/1859) меняет Next.js-плагин так, чтобы он повторял правило самого Next.js для router directories. Если в корне есть `app/` или `pages/`, соответствующая папка внутри `src/` игнорируется Next.js и теперь не должна автоматически становиться production entry в Knip.

До исправления проект мог выглядеть так:

```txt
pages/home.tsx           # реальный route
src/pages/HomePage.tsx   # импортируется из route
src/pages/OldPage.tsx    # никем не импортируется
```

Next.js в таком проекте использует root `pages/` и игнорирует `src/pages`, но старый Knip добавлял оба паттерна и тем самым скрывал `src/pages/OldPage.tsx` от отчёта unused files. В `6.25.0` helper `getRouterDirPrefix` проверяет `app` и `pages` независимо: root-папка выигрывает, `src/` используется только если root-папки нет, а когда ни одной папки нет — Knip оставляет прежний широкий fallback `{,src/}`.

Root-level file conventions (`middleware`, `proxy`, `instrumentation`, `instrumentation-client`) остаются в обоих местах, потому что это отдельная часть поведения Next.js-плагина.

## Bun: тестовые entrypoints включаются только для реального `bun test`

PR [#1860](https://github.com/webpro-nl/knip/pull/1860) переписывает Bun-плагин вокруг нового parser'а shell-команд на базе `unbash`. Плагин теперь включается, если найден `bun.lock`/`bun.lockb` или если scripts реально содержат `bun test`.

Поддержаны и варианты, где `test` идёт после опций:

```json
{
  "scripts": {
    "test": "bun --config=./bunfig.toml test",
    "test:unit": "bun test src/unit"
  }
}
```

Когда `bun test` запускается без явных targets, Knip использует дефолтные Bun-паттерны:

```txt
**/*.{test,spec}.{js,jsx,ts,tsx}
**/*_{test,spec}.{js,jsx,ts,tsx}
```

А если targets указаны, директория разворачивается в эти паттерны внутри неё, файл остаётся файлом, а glob остаётся glob'ом. Плагин также добавляет `toIgnore("bun", "dependencies")`, чтобы dependency `bun` не считалась unused только потому, что сам plugin предоставил стандартные entries.

Новый общий helper `getScriptCommands` из `packages/knip/src/util/scripts.ts` важен не только для Bun: он обходит `AndOr`, `Pipeline`, `If`, `While`, `For`, subshell и функции, а для обёрток вроде `cross-env` и `retry-cli` рекурсивно разбирает фактическую вложенную команду.

## Новые и расширенные плагины: Fumadocs, eve, Storybook MDX, TypeDoc, GraphQL Codegen

В `6.25.0` появились два новых plugin-level сценария.

Первый — Fumadocs. Коммит [`77c2142`](https://github.com/webpro-nl/knip/commit/77c2142fcc6ad6fdee651c5dc60a74c158cd8de3) добавляет плагин, который включается при наличии `fumadocs-core`, `fumadocs-mdx` или `fumadocs-ui`, считает entry `source.config.{js,ts,mjs}` и `content/**/*.mdx`, а при зависимости `fumadocs-mdx` регистрирует общий MDX compiler:

```ts
const entry = ["source.config.{js,ts,mjs}", "content/**/*.mdx"];
```

Второй — eve. PR [#1864](https://github.com/webpro-nl/knip/pull/1864) добавляет плагин для Vercel Eve, где capabilities обнаруживаются по файловой структуре, а не через центральный import registry. Плагин помечает production entries для root/agent layouts, runtime slots и вложенных subagents:

```txt
agent.{js,jsx,ts,tsx,mjs,cjs,mts,cts}
agent/tools/**/*.{js,jsx,ts,tsx,mjs,cjs,mts,cts}
agent/skills/**/*.{js,jsx,ts,tsx,mjs,cjs,mts,cts}
agent/subagents/**/agent.{js,jsx,ts,tsx,mjs,cjs,mts,cts}
agent/subagents/**/{connections,hooks,skills,tools}/**/*.{js,jsx,ts,tsx,mjs,cjs,mts,cts}
```

Фикстура специально содержит `agent/lib/unused.ts`: плагин не делает весь `agent/` entry и поэтому всё ещё позволяет находить реально мёртвые helper-файлы.

Ещё несколько плагинов получили точечные улучшения:

- Storybook регистрирует MDX compiler, если есть `@storybook/addon-docs` или `@storybook/addon-essentials`, поэтому импорты из `.mdx` docs не превращают helpers в unused exports/files.
- TypeDoc теперь считает `customCss` и `customJs` entry-файлами относительно файла конфигурации.
- GraphQL Codegen помечает outputs из `generates` как production entries: файл остаётся файлом, а output-директория превращается в `<dir>**`. Это защищает сгенерированные exports от ложного unused.
- Jest теперь разбирает inline Babel config внутри `transform: { ...: ["babel-jest", { presets, plugins }] }` тем же механизмом, что и Babel-плагин.
- Prettier читает `plugins` не только в корне конфига, но и внутри `overrides[].options.plugins`.
- Serverless Framework учитывает esbuild `inject` из `custom.esbuild` и `build.esbuild` как production entries и по-прежнему добавляет optional dependency `esbuild`.

## TypeScript и module graph: меньше false positive'ов на aliases и augmentation

Коммит [`9cc5a41`](https://github.com/webpro-nl/knip/commit/9cc5a4132793990b5fcf9eae6f3db37426d979c0) продолжает исправление module augmentation из предыдущего релиза, но теперь для path aliases. Если augment-модуль задан не относительным путём, а alias'ом, Knip резолвит его к внутреннему модулю и регистрирует использованные типы как type imports.

```ts
declare module "#events" {
  type EntityWithMeta = BaseEntity & { meta: string };

  export interface EventBusEvents extends EventEnvelope {
    "entity:created": EntityWithMeta;
  }
}
```

Если `#events` указывает на внутренний `events.ts`, то `BaseEntity` и `EventEnvelope` теперь считаются использованными exports этого модуля. Внешние и ambient augmentations, например `declare module "@vue/runtime-core"`, остаются нейтральными: Knip не должен на основании такой декларации засчитывать dependency usage или создавать unresolved import.

Коммит [`e3f0e5c`](https://github.com/webpro-nl/knip/commit/e3f0e5c69641bb7dfab06585251c98e3b3def049) исправляет ещё одну alias-ситуацию: inherited `tsconfig.paths` теперь резолвятся от директории extended-конфига, а не от директории дочернего файла. Это важно для generated config'ов вроде `.svelte-kit/tsconfig.json`, которые расширяют конфиг из другой папки.

Коммит [`9422d06`](https://github.com/webpro-nl/knip/commit/9422d06adb07fd29a3388ee6b037cda47fe28d1a) добавляет обработку var interpolations в shell command scripts. В связке с новым command walker'ом это уменьшает число случаев, где зависимость или binary спрятаны внутри более сложного script'а и раньше не попадали в анализ.

## Что это значит при обновлении

Для большинства проектов `6.25.0` — безопасный минор: новые проверки либо opt-in (`cycles`), либо уменьшают ложные срабатывания в уже поддержанных интеграциях. На что стоит обратить внимание после обновления:

```bash
pnpm add -D knip@6.25.0
pnpm knip
pnpm knip --cycles --reporter cycles
```

- Если хотите проверять циклические зависимости, включите `cycles` явно и решите, будет ли это `warn` или `error`.
- Если в проекте есть `packageExtensions` для Yarn/pnpm, часть прежних `unused dependencies` по peer-пакетам может исчезнуть.
- В Next.js-проектах с одновременными root `pages/`/`app/` и `src/pages`/`src/app` Knip может начать показывать файлы из игнорируемой Next.js директории как unused — это ожидаемое поведение.
- Для Fumadocs, eve, Storybook MDX, TypeDoc assets, GraphQL Codegen outputs, Serverless esbuild inject и Bun tests должно стать меньше ручных `entry`, `project` и `ignore` в `knip.json`.
