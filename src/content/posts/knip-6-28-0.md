Вышел минорный релиз [Knip 6.28.0](https://github.com/webpro-nl/knip/releases/tag/knip%406.28.0) — инструмента для поиска неиспользуемых файлов, зависимостей и exports в JavaScript- и TypeScript-проектах. В нём заметно расширен анализ конфигураций и CLI-команд: Knip понимает `markdownlint-cli2`, `tsdown`, Tailwind CLI и настраиваемый каталог функций Convex. Одновременно релиз уточняет, что считать публичным export, и убирает несколько источников ложных срабатываний.

Источники: [GitHub Release](https://github.com/webpro-nl/knip/releases/tag/knip%406.28.0), [compare 6.27.0…6.28.0](https://github.com/webpro-nl/knip/compare/knip%406.27.0...knip%406.28.0), коммиты [markdownlint-cli2](https://github.com/webpro-nl/knip/commit/323b96f55b38b53f5e4648d72fecd968fae35147), [tsdown](https://github.com/webpro-nl/knip/commit/0d47a45bfc518a7bf73a4b4878aeab236d6f9791), [Convex](https://github.com/webpro-nl/knip/commit/e01f21c44773186e7ad74e7badf9fc81b82edfb5), [JSDoc](https://github.com/webpro-nl/knip/commit/6ae48aadbe03952690c8608e6df42bd039cf6624), [re-exports](https://github.com/webpro-nl/knip/commit/a113a5f4eedc81ef56b00a3e488cc94febefede5) и [preprocessors](https://github.com/webpro-nl/knip/commit/844358367e9d1939bbf47f8dcb97d810741f56c1). Формат версии — `6.28.0`, то есть это qualifying minor-релиз; `featured` оставлен `false`.

## `markdownlint-cli2`: конфигурация и подключаемые модули больше не теряются

Встроенный plugin `markdownlint` теперь включается как при `markdownlint-cli`, так и при `markdownlint-cli2`. Помимо прежних `.markdownlint.{json,jsonc,yml,yaml}` он читает `.markdownlint-cli2.{jsonc,yaml,cjs,mjs}` и JS-конфиги с теми же расширениями.

Для `markdownlint-cli2` важна другая форма настроек: `extends` может лежать внутри `config`, а дополнительные пакеты объявляются в `customRules`, `markdownItPlugins` и `outputFormatters`. Knip извлекает имена из этих полей и связывает их с dependency graph; встроенные `markdownlint/style/*` и `markdownlint-cli2-formatter-default` намеренно не считаются отдельными npm-зависимостями.

```jsonc
// .markdownlint-cli2.jsonc
{
  "config": {
    "extends": "markdownlint-config-prettier",
  },
  "customRules": ["markdownlint-rule-titlecase"],
  "markdownItPlugins": [["markdown-it-anchor", { "permalink": true }]],
  "outputFormatters": [
    ["markdownlint-cli2-formatter-summarize"],
    ["markdownlint-cli2-formatter-default"],
  ],
}
```

Также plugin ищет правила, переданные скрипту через `--rules` или `-r`, и распознаёт сам вызов `markdownlint-cli2`. Поэтому зависимости из такого конфига не должны попадать в `unused dependencies` только потому, что они загружаются markdownlint'ом, а не импортируются приложением.

## `tsdown` и Tailwind CLI: entrypoints берутся из команды сборки

Плагин `tsdown` получил разбор CLI-аргументов. Positional arguments и `--entry` становятся production entries, а `--external`, `--deps.never-bundle` / `--deps.neverBundle` — optional dependency references. Флаги `--publint` и `--attw` также связывают соответственно `publint` и `@arethetypeswrong/core`.

```json
{
  "scripts": {
    "build": "tsdown --root src --platform node --publint src/library.ts src/modules/nested.ts --dts --no-config --outDir dist"
  }
}
```

В таком сценарии `src/library.ts` и `src/modules/nested.ts` считаются входами, а не неиспользуемыми файлами; `src/orphan.ts` без связи с командой всё ещё может быть показан в отчёте. Обработчик специально не принимает за entry значение `inline` или `hidden` после `--sourcemap`, путь после `--watch`, а также подкоманды `tsdown create` и `tsdown migrate`.

Похожее точечное улучшение появилось для Tailwind CSS CLI: если в scripts есть `tailwindcss -i ./src/theme.css -o ./dist/app.css`, plugin извлекает `-i` как alias `--input` и регистрирует `src/theme.css` production entry. Это покрывает распространённую Tailwind 4-конфигурацию, где stylesheet с `@import "tailwindcss"` является корнем CSS-графа.

## Convex читает `functions` из `convex.json`

Раньше Convex plugin ориентировался на каталог `convex/`. Теперь он загружает `convex.json`, берёт строковое поле `functions` и добавляет `**/*.{js,ts}` из указанного каталога как production entries. Если поля нет, дефолт `convex` сохраняется.

```json
{
  "functions": "backend/"
}
```

При такой конфигурации `backend/http.ts` с `httpRouter()` учитывается как runtime-модуль Convex, а файл в старом `convex/`, который не связан с настройкой, остаётся кандидатом на `unused files`. Это важно для приложений, которые перенесли backend в нестандартную папку.

## JSDoc `@import` теперь удерживает типы в графе

Knip научился разбирать `@import` в JSDoc через parser, а не регулярным выражением. Поддержаны default import, named import, alias и namespace import; все они регистрируются как type-only references. Например, в следующем коде `DefaultType`, `NamedType`, `AliasedType` и `types.NamespaceType` не должны считаться неиспользуемыми exports:

```js
/**
 * @import DefaultType, {
 *   NamedType,
 *   AliasedType as LocalType,
 * } from "./types.ts"
 * @import * as types from "./namespace-types.ts"
 */

/** @param {DefaultType} value @param {types.NamespaceType} ns */
export function useTypes(value, ns) {
  return { value, ns };
}
```

При этом Knip проверяет, что `@import` действительно начинается с JSDoc-тега, а не просто упомянут в тексте комментария. Неиспользуемый `UnusedType` в исходном модуле продолжает попадать в отчёт.

## Граница публичного API: re-export из entry сохраняет члены enum и namespace

Исправление для `ignoreExportsUsedInFile` затрагивает библиотеки, где public API проходит через entry-файл. Если namespace или enum re-export'ится из entry и `includeEntryExports` выключен, его члены сохраняются публичными даже тогда, когда внутренний код использует только часть из них.

```ts
// index.ts — package entry
export { Mode } from "./mode";

// mode.ts
export enum Mode {
  internal = "internal",
  external = "external",
}
```

В обычном режиме `Mode.external` не будет ошибочно объявлен неиспользуемым: его может использовать потребитель пакета. Если же явно включить `includeEntryExports`, Knip снова имеет право отчитываться о публичных exports. Обычный import внутри entry не считается re-export и не получает это послабление.

Отдельно исправлена классификация re-exports при `ignoreExportsUsedInFile`: re-export сам по себе больше не делает символ автоматически «использованным». Если модуль экспортирует `grape` и `plum`, но в приложении реально используется только `grape`, `plum` остаётся в issues и в barrel, и в исходном модуле.

## Цепочка preprocessors, `node:` и правила `.gitignore`

Несколько исправлений делают результаты анализа предсказуемее:

- несколько повторов `--preprocessor` теперь выполняются строго по порядку и каждый получает результат предыдущего, включая async preprocessor; раньше второй обработчик мог получить вместо options загруженный модуль и завершиться с `id.startsWith is not a function`;
- любой specifier с префиксом `node:` считается built-in, даже если текущая версия Node ещё не знает модуль. Поэтому `node:sqlite` и `node:some-future-builtin` не становятся `unlisted dependencies`;
- `.gitignore`-правило `generated/*` теперь скрывает и вложенные файлы. Отрицание `!generated/keep` возвращает только разрешённую ветку и не раскрывает, например, `generated/other/deep.ts`;
- при наличии negations Knip повторно сверяет результаты glob cache с gitignore-обработчиком: исключение `!lib/dist/` не должно «разигнорировать» соседний root `dist/`.

## Nuxt/Nitro: меньше ложных unresolved и точнее modules

Nuxt plugin игнорирует внутренние виртуальные импорты `#build/*`, `#internal/*`, `#spa-template`, `#components` и `#imports` как `unresolved`. Для Nitro аналогично исключаются `#nitro-internal-*`.

Одновременно `srcDir` приводится к абсолютному пути относительно `nuxt.config.*`; local modules и layers из `extends` резолвятся от реального конфигурационного каталога. Модули-пакеты из `modules` теперь отмечаются как production dependencies — в strict production-анализе Knip корректно обнаружит незаявленный `nuxt-module`, но не перепутает его с локальным модулем.

## Обновление

```bash
pnpm add -D knip@6.28.0
pnpm knip
```

После обновления имеет смысл убрать ручные `entry` для `tsdown`, Tailwind CLI и Convex functions только после проверки собственного отчёта. Проектам с public enum/namespace exports стоит особенно внимательно сравнить вывод с `includeEntryExports`: новая логика по умолчанию защищает API пакета, а включённый флаг намеренно возвращает строгую проверку публичных символов.
