[`pi`](https://github.com/earendil-works/pi) выпустил минорный релиз [`v0.81.0`](https://github.com/earendil-works/pi/releases/tag/v0.81.0). В нём локальный `llama.cpp` router стал встроенным провайдером с управлением моделями из TUI, расширения получили API для регистрации полного `pi-ai` provider, а статистика сессии начала учитывать не только ответы агента, но и инструменты с суммаризацией контекста.

Источники: GitHub Release [`earendil-works/pi@v0.81.0`](https://github.com/earendil-works/pi/releases/tag/v0.81.0), compare [`v0.80.0...v0.81.0`](https://github.com/earendil-works/pi/compare/v0.80.0...v0.81.0), документация релизного тега для [llama.cpp](https://github.com/earendil-works/pi/blob/v0.81.0/packages/coding-agent/docs/llama-cpp.md), [кастомных провайдеров](https://github.com/earendil-works/pi/blob/v0.81.0/packages/coding-agent/docs/custom-provider.md), [providers](https://github.com/earendil-works/pi/blob/v0.81.0/packages/coding-agent/docs/providers.md) и [compaction](https://github.com/earendil-works/pi/blob/v0.81.0/packages/coding-agent/docs/compaction.md), а также PR [#6858](https://github.com/earendil-works/pi/pull/6858), [#6865](https://github.com/earendil-works/pi/pull/6865), [#6772](https://github.com/earendil-works/pi/pull/6772), [#6671](https://github.com/earendil-works/pi/pull/6671), [#6864](https://github.com/earendil-works/pi/pull/6864) и [#6854](https://github.com/earendil-works/pi/pull/6854).

## Локальные GGUF-модели через llama.cpp router

Pi теперь умеет работать именно с router-режимом `llama-server`: сервер обнаруживает несколько GGUF-моделей и загружает или выгружает их по запросу. Это не режим одного `--model`: документация подчёркивает, что для router нельзя передавать `--model`, `-m` или `-hf`.

Минимальный пример запуска сервера с каталогом моделей:

```bash
llama-server \
  --models-dir ~/models \
  --no-models-autoload \
  --jinja \
  --host 127.0.0.1 \
  --port 8080 \
  -ngl 999 \
  -c 32768
```

`--no-models-autoload` оставляет загрузку явной через Pi; `--jinja` включает совместимые chat templates и tool calling, а `-ngl 999` переносит на GPU максимально возможное число слоёв. После запуска соединение настраивается интерактивно:

```text
/login llama.cpp
```

По умолчанию Pi ожидает `http://127.0.0.1:8080`; то же можно задать без TUI через `LLAMA_BASE_URL` и необязательный `LLAMA_API_KEY`:

```bash
export LLAMA_BASE_URL=http://127.0.0.1:8080
export LLAMA_API_KEY=optional-secret
pi
```

Команда `/llama` показывает фактическое состояние router и позволяет выбрать выгруженную модель для загрузки, а загруженную — для выгрузки. Через **Download model…** можно искать на Hugging Face, выбрать репозиторий и квантование либо ввести `owner/repository[:quant]` вручную. При загрузке и скачивании показывается прогресс; Escape вызывает подтверждение отмены. Pi не удаляет файлы моделей и не выгружает другие модели молча: при занятом router он предлагает выгрузить их или оставить загруженными.

Это важное разделение двух шагов: только загруженные модели появляются в `/model`; после `/llama` нужно отдельно выбрать нужную модель для текущей сессии через `/model`. Для gated-репозиториев токен ищется сначала в `HF_TOKEN`, затем в стандартных путях Hugging Face; сам download выполняет процесс `llama-server`, поэтому ему тоже нужен `HF_TOKEN`.

## Расширение может зарегистрировать полный `pi-ai` Provider

В 0.80.0 Pi уже перешёл к provider-oriented `Models` API. Теперь extension API принимает не только старую конфигурацию endpoint/models, но и готовый объект `Provider`, созданный, например, через `createProvider()`. Это путь для собственного login flow, обновляемого каталога, фильтрации моделей и нестандартного streaming-протокола.

Ниже минимальный пример встроенной аутентификации для локального OpenAI-compatible endpoint из документации релиза:

```ts
import { createProvider, openAICompletionsApi } from "@earendil-works/pi-ai";
import type { ExtensionAPI } from "@earendil-works/pi-coding-agent";

export default function (pi: ExtensionAPI) {
  pi.registerProvider(
    createProvider({
      id: "native-local",
      name: "Native Local",
      baseUrl: "http://localhost:8080/v1",
      auth: {
        apiKey: {
          name: "Local server API key",
          async login(interaction) {
            return {
              type: "api_key",
              key: await interaction.prompt({
                type: "secret",
                message: "API key",
              }),
            };
          },
          async resolve({ credential }) {
            return credential?.key
              ? { auth: { apiKey: credential.key }, source: "stored API key" }
              : undefined;
          },
        },
      },
      models: [],
      api: openAICompletionsApi(),
    })
  );
}
```

Для каталога, который надо получить по сети, factory расширения может быть `async`: Pi ждёт её завершения до interactive startup и `pi --list-models`. В старой конфигурационной форме `pi.registerProvider("my-provider", { models: [...] })` список `models` по-прежнему работает, но **заменяет** все существующие модели этого provider. `pi.unregisterProvider(name)` теперь снимает динамические модели, API-key fallback, OAuth-регистрацию и custom stream handlers и возвращает перекрытые built-ins.

## Qwen Token Plan: два региональных провайдера

PR [#6858](https://github.com/earendil-works/pi/pull/6858) добавил в built-in catalog два API-key provider:

| Provider id          | Регион         | Переменная ключа             | Endpoint                                      |
| -------------------- | -------------- | ---------------------------- | --------------------------------------------- |
| `qwen-token-plan`    | international  | `QWEN_TOKEN_PLAN_API_KEY`    | `token-plan.ap-southeast-1.maas.aliyuncs.com` |
| `qwen-token-plan-cn` | mainland China | `QWEN_TOKEN_PLAN_CN_API_KEY` | `token-plan.cn-beijing.maas.aliyuncs.com`     |

Оба используют OpenAI-compatible API и каталог из 15 текстовых моделей Qwen, DeepSeek, GLM, Kimi и MiniMax; в нём включена поддержка DashScope `enable_thinking`. Фактический доступ зависит от плана: автор PR отдельно отмечает, что personal tier открывает шесть из пятнадцати моделей, а остальные могут вернуть `403` как team-tier only. Поэтому наличие модели в каталоге не гарантирует подписку на неё.

Ключ можно сохранить через `/login` либо передать окружением:

```bash
export QWEN_TOKEN_PLAN_API_KEY=sk-sp-...
pi
```

Заодно обработчик переполнения контекста распознаёт ответ DashScope вида `Range of input length should be [1, X]`, который приходит обычной ошибкой, а не `finish_reason: "length"`.

## RPC узнаёт доступные уровни thinking

Для интеграций в RPC mode добавлены команда `get_available_thinking_levels` и метод `RpcClient.getAvailableThinkingLevels()`. Команда не заставляет клиента гадать по каталогу модели: она спрашивает текущую сессию и возвращает допустимый набор. Для модели без reasoning поддержка ограничивается `["off"]`.

```json
{ "type": "get_available_thinking_levels" }
```

Типичный ответ для reasoning-модели:

```json
{
  "type": "response",
  "command": "get_available_thinking_levels",
  "success": true,
  "data": {
    "levels": ["off", "minimal", "low", "medium", "high"]
  }
}
```

В package root также экспортированы типы lifecycle-событий: `MessageStartEvent`, `MessageUpdateEvent`, `MessageEndEvent`, `ToolExecutionStartEvent`, `ToolExecutionUpdateEvent` и `ToolExecutionEndEvent`. Авторы интеграций теперь могут импортировать их напрямую из `@earendil-works/pi-coding-agent`, а не из внутреннего пути.

## Usage сессии включает инструменты и суммаризацию

До этого релиза расход токенов и стоимость в session totals не полностью отражали работу, которая происходит вне обычного assistant response. PR [#6671](https://github.com/earendil-works/pi/pull/6671) добавляет поле `usage` в сохранённые `CompactionEntry` и `BranchSummaryEntry`, а также в результаты выполнения инструментов.

Практический эффект: footer totals, session statistics и сохранённая сессия учитывают вызовы LLM для `/compact` и переходов через `/tree`, а также usage инструментов. При этом PR явно фиксирует границу API: usage tool result пока сохраняется, но обработчики `tool_result` не получают и не могут изменить это поле — для этого пришлось бы расширять базовые event-типы.

## Исправления, которые заметны при смене provider и после обновления

- При переключении с OpenAI Responses-style модели на `openai-completions` Pi нормализует replayed tool-call IDs. Раньше два вызова с общим `call_id` и разными суффиксами item могли превратиться в одинаковый ID и получить `400` от следующего Chat Completions request; PR [#6854](https://github.com/earendil-works/pi/pull/6854) сохраняет уникальность ID.
- Сохранённые API-key credentials теперь применяют свой объект `env`, а не всегда падают обратно к `process.env`. Это исправляет, в частности, Azure OpenAI и Bedrock-профили с `AZURE_OPENAI_*`, `AWS_REGION`, proxy-настройками или `PI_CACHE_RETENTION` в `auth.json` ([#6864](https://github.com/earendil-works/pi/pull/6864)).
- Сетевое обновление model catalog вынесли из startup initialization в уже работающие interactive и RPC modes. Одновременно обновлённый bundled catalog больше не должен перезаписываться старым persisted remote catalog после апгрейда.
- Раннее окончание OpenAI Responses stream теперь инициирует автоматический retry вместо преждевременного завершения agent run; а сообщения, пришедшие во время compaction, сохраняют порядок доставки steering и follow-up.

Версия 0.81.0 особенно полезна тем, кто запускает Pi локально или в корпоративной инфраструктуре: `llama.cpp` router теперь управляется из знакомого TUI, extension может стать настоящим provider с собственной аутентификацией и transport-логикой, а отчётность по сессии перестаёт скрывать стоимость compaction и branch summaries.
