Knip выпустил минорный релиз [`6.23.0`](https://github.com/webpro-nl/knip/releases/tag/knip%406.23.0). Это обновление особенно заметно для Astro-проектов и JavaScript-инфраструктуры вокруг Vite/Vitest: Knip научился видеть больше реальных production entrypoints и меньше шуметь на зависимостях, которые конфигурации используют как hints для бандлера.

Источники для обзора — GitHub Release [`webpro-nl/knip@knip@6.23.0`](https://github.com/webpro-nl/knip/releases/tag/knip%406.23.0), PR [#1828](https://github.com/webpro-nl/knip/pull/1828), [#1829](https://github.com/webpro-nl/knip/pull/1829), [#1830](https://github.com/webpro-nl/knip/pull/1830) и коммиты [`8a6050e`](https://github.com/webpro-nl/knip/commit/8a6050e6a92da81d4875f730f852fb7d9252a018), [`849b5ac`](https://github.com/webpro-nl/knip/commit/849b5ac230e7a8c103b6e1b1e2ddb333d2da3ca0). Релиз минорный, поэтому `featured: false`.

## Starlight: `customCss` теперь считается production entry

PR [#1828](https://github.com/webpro-nl/knip/pull/1828) расширяет существующий плагин Starlight. До `6.23.0` Knip уже умел находить component overrides Starlight, но не учитывал CSS-файлы, перечисленные в опции `customCss`. В проектах на Starlight с Tailwind или отдельными stylesheet'ами это могло приводить к ложным отчётам об unused files.

В `packages/knip/src/plugins/starlight/index.ts` плагин теперь:

- строит import map для `astro.config.*`;
- находит default import из `@astrojs/starlight`;
- достаёт аргумент вызова `starlight(...)` через `findCallArg`;
- читает свойство `customCss` через `getPropertyValues`;
- добавляет каждый путь как `toProductionEntry(id)`.

Поддерживаемый сценарий из regression fixture выглядит так:

```js
import starlight from "@astrojs/starlight";
import { defineConfig } from "astro/config";

export default defineConfig({
  integrations: [
    starlight({
      customCss: ["./styles/used.css"],
    }),
  ],
});
```

Практический результат: `./styles/used.css` становится production entry для анализа Knip. В тесте для fixture `starlight/custom-css` ожидаются `files: 1` и `dependencies: 1`, а соседний `styles/unused.css` остаётся кандидатом на отчёт как реально неиспользуемый файл.

## Новый плагин `astro-markdoc`

PR [#1829](https://github.com/webpro-nl/knip/pull/1829) добавляет отдельный Knip-плагин для [`@astrojs/markdoc`](https://docs.astro.build/en/guides/integrations-guide/markdoc/). Он включается, когда в зависимостях проекта есть `@astrojs/markdoc`, и ищет конфигурации по шаблону:

```txt
markdoc.config.{js,ts,mjs,mts}
```

В публичной конфигурации Knip новый плагин называется `astro-markdoc`: это имя добавлено в `src/plugins/index.ts`, `src/schema/plugins.ts`, `src/types/PluginNames.ts` и `schema.json`. Значит, его можно настраивать так же, как остальные плагины Knip:

```json
{
  "astro-markdoc": {
    "config": ["markdoc.config.mjs"]
  }
}
```

Главная логика находится в новом `packages/knip/src/plugins/astro-markdoc/index.ts`. Плагин загружает local Markdoc config и просматривает `nodes` и `tags`; если внутри элемента есть поле `render`, Knip извлекает путь к Astro-компоненту и превращает его в production entry.

Regression fixture показывает типичный Astro Markdoc config:

```js
import { defineMarkdocConfig, component } from "@astrojs/markdoc/config";

export default defineMarkdocConfig({
  nodes: {
    heading: {
      render: component("./src/components/Heading.astro"),
    },
  },
  tags: {
    callout: {
      render: component("./src/components/Callout.astro", "CTA"),
    },
  },
});
```

Внутренний тип `MarkdocRenderSpecifier` покрывает не только строки, но и объектные формы. Если `render` — строка, Knip отбрасывает суффикс после `#`; если это объект, проверяются строковые поля `Component`, `file` и `path`. Поэтому компонентный specifier вроде `./src/components/Callout.astro#CTA` учитывается как путь к файлу `./src/components/Callout.astro`, а не как отдельный пакет.

Для пользователей Astro это важное уменьшение false positive'ов: компоненты, подключённые только через Markdoc `nodes` или `tags`, больше не должны выглядеть «мёртвыми» для Knip.

## Vite/Vitest включаются при наличии `vite-plus`

Исправление [#1830](https://github.com/webpro-nl/knip/pull/1830) адресует проекты, где вместо прямой зависимости на `vite` или `vitest` используется пакет `vite-plus`. В `6.23.0` Knip добавил `vite-plus` в списки enablers сразу двух плагинов:

```ts
// packages/knip/src/plugins/vite/index.ts
const enablers = ["vite", "vitest", "vite-plus"];

// packages/knip/src/plugins/vitest/index.ts
const enablers = ["vitest", "vite-plus"];
```

Новый тест `test/plugins/vite-plus.test.ts` проверяет именно это поведение: при `devDependencies: { "vite-plus": "*" }` в fixture Knip должен включить и `vite`, и `vitest` в `enabledPlugins['.']`.

Для монорепозиториев это значит, что файлы вроде `vite.config.*`, `vitest.config.*`, `import.meta.glob(...)` и Vitest workspace config будут анализироваться даже там, где прямой зависимости на `vite`/`vitest` нет, но есть `vite-plus`.

## Поддержка runtime/package manager `nub`

Коммит [`8a6050e`](https://github.com/webpro-nl/knip/commit/8a6050e6a92da81d4875f730f852fb7d9252a018) добавляет resolver для `nub` и учитывает `nub --test` в node test runner plugin.

Изменения состоят из нескольких частей:

- в `src/binaries/resolvers/index.ts` появились resolver'ы для `nub` и `nubx`;
- `nub` и `nubx` добавлены в `IGNORED_GLOBAL_BINARIES`, чтобы сами runtime-команды не репортились как project dependency;
- новый `src/binaries/resolvers/nub.ts` разбирает команды `x`, `dlx`, `exec`, `run`, `watch`, package-manager команды вроде `install`, `add`, `remove`, `why`, а также fallback-сценарий запуска через Node;
- node plugin теперь считает тестовым скриптом не только `node ... --test`, но и `nub ... --test`.

Regression tests фиксируют конкретные сценарии разбора scripts:

```json
{
  "scripts": {
    "test": "nub --test",
    "dev": "nub watch ./main.ts",
    "lint": "nubx eslint . --fix"
  }
}
```

Для `nubx eslint . --fix` Knip должен увидеть binary `eslint`, для `nub watch ./main.ts` — entry `main.ts`, а для `nub --test --test-reporter=custom-reporter` — Node test runner и reporter dependency `custom-reporter`. Отдельный fixture `node-test-runner-nub` проверяет production-режим: тестовый файл `index.test.js` исключается из production counters, как и при обычном `node --test`.

## Меньше ложных `unlisted` в Vitest config

Коммит [`849b5ac`](https://github.com/webpro-nl/knip/commit/849b5ac230e7a8c103b6e1b1e2ddb333d2da3ca0) меняет обработку зависимостей из Vitest/Vite config. Раньше Knip превращал значения `resolve.dedupe` и `optimizeDeps.include` в обычные dependency inputs, из-за чего такие пакеты могли попадать в `issues.unlisted`.

В `packages/knip/src/plugins/vitest/index.ts` оба источника теперь помечаются как optional:

```ts
for (const dependency of cfg.resolve?.dedupe ?? []) {
  inputs.add(toDependency(dependency, { optional: true }));
}

for (const dependency of cfg.optimizeDeps?.include ?? []) {
  const packageName = getOptimizeDepsIncludePackageName(dependency);
  if (packageName) inputs.add(toDependency(packageName, { optional: true }));
}
```

Тестовый fixture `vitest9` добавил сразу три проверки:

```ts
assert(!issues.unlisted["vitest.config.ts"]?.["top-level-dep"]);
assert(!issues.unlisted["vitest.config.ts"]?.["project-level-dep"]);
assert(!issues.unlisted["vitest.config.ts"]?.["project-dedupe-dep"]);
```

Это не означает, что Knip перестаёт читать `optimizeDeps` или `dedupe`: он всё ещё учитывает эти specifier'ы для графа анализа. Но они больше не трактуются как обязательные зависимости `package.json`, потому что в Vite/Vitest это часто hints для предварительной оптимизации или дедупликации, а не прямой импорт приложения.

## Кому стоит обновиться

`6.23.0` стоит поставить в первую очередь командам с Astro-документацией и сайтами на Starlight/Markdoc: Knip теперь лучше понимает файлы, которые подключаются через framework config, а не через обычные `import` в исходниках.

Обновление также полезно проектам на `vite-plus`, пользователям `nub` и монорепозиториям с Vitest workspace config. В этих случаях релиз уменьшает ручную настройку Knip: меньше `ignore` для реально используемых CSS/компонентов, меньше ложных `unlisted` для Vite optimize hints и более точное включение встроенных плагинов.

Полный список изменений доступен в [GitHub Release 6.23.0](https://github.com/webpro-nl/knip/releases/tag/knip%406.23.0).
