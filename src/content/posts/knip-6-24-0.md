Knip выпустил минорный релиз [`6.24.0`](https://github.com/webpro-nl/knip/releases/tag/knip%406.24.0). Это обновление продолжает линию предыдущих релизов: меньше ручных `ignore`, больше framework-aware анализа и меньше false positive'ов в проектах, где зависимости используются через конфиги, CSS-препроцессоры, JSDoc или TypeScript module augmentation.

Источники для обзора — GitHub Release [`webpro-nl/knip@knip@6.24.0`](https://github.com/webpro-nl/knip/releases/tag/knip%406.24.0), compare [`knip@6.23.0...knip@6.24.0`](https://github.com/webpro-nl/knip/compare/knip@6.23.0...knip@6.24.0), PR [#1844](https://github.com/webpro-nl/knip/pull/1844), [#1845](https://github.com/webpro-nl/knip/pull/1845), [#1846](https://github.com/webpro-nl/knip/pull/1846), [#1848](https://github.com/webpro-nl/knip/pull/1848) и коммиты [`aea7923`](https://github.com/webpro-nl/knip/commit/aea7923438f0a8f459458578526f358b02497f77), [`7901abd`](https://github.com/webpro-nl/knip/commit/7901abd3c4b496f212445bff9768efc9548ded61), [`260f192`](https://github.com/webpro-nl/knip/commit/260f19230f42c9dceaac97d55bf34d17902c5b38), [`bb0eeb6`](https://github.com/webpro-nl/knip/commit/bb0eeb6e33d050dab736134b5b692a3d6413f358). Релиз минорный, поэтому `featured: false`.

## Less и Stylus стали встроенными компиляторами

Главное пользовательское изменение в PR [#1845](https://github.com/webpro-nl/knip/pull/1845) — Knip теперь умеет извлекать зависимости не только из Sass/SCSS, но и из Less и Stylus. В документации `packages/docs/src/content/docs/features/compilers.md` к списку built-in compilers добавлены:

```txt
.less
.styl + .stylus
```

В коде появились новые компиляторы `packages/knip/src/compilers/less.ts` и `packages/knip/src/compilers/stylus.ts`. Less-компилятор ищет `@import`, поддерживает обычные строки и `url(...)`, отбрасывает внешние URL (`//`, `http://`, `https://`) и разворачивает относительные пути без расширения в кандидат с `.less`:

```less
@import "variables";
@import "./theme.less";
@import (reference) "./mixins";
@import (less) "./legacy.css";
@import url("./base");
@import "https://fonts.googleapis.com/css?family=Roboto";
```

Из этого Knip должен учитывать локальные `variables.less`, `theme.less`, `mixins.less`, `legacy.css` и `base.less`, но не внешний Google Fonts URL. Для package imports покрыты и привычные Less-формы с тильдой:

```less
@import "~bootstrap/less/bootstrap.less";
@import "@fontsource/lato/less/mixins.less";
@import "~@ant-design/pro-components/lib/index.less";
@import "foundation-sites/less/foundation";
```

Stylus-компилятор аналогично читает `@import` и `@require`, понимает `.styl`, `.stylus`, `.css`, а для относительного пути без расширения проверяет два кандидата: `<name>.styl` и `<name>/index.styl`. Regression fixture покрывает package imports вроде:

```stylus
@require '~nib/lib/nib.styl'
@require '@stylus-plugins/typography/lib/index.styl'
@require '~rupture/stylus/rupture.styl'
@require 'foundation-sites/stylus/foundation'
@require './variables'
```

Практический результат: если проект реально использует `less` или `stylus`, Knip может найти файлы и package dependencies, которые раньше выглядели неиспользуемыми, потому что были видны только из stylesheet'ов.

## Astro, Svelte и Vue теперь читают импорты из `<style lang="...">`

В том же PR [#1845](https://github.com/webpro-nl/knip/pull/1845) Less/Stylus подключены не только как отдельные файлы. Новый helper `stylePreprocessorImports` в `packages/knip/src/compilers/style-preprocessors.ts` проходит по `<style>`-блокам, смотрит атрибут `lang` и направляет тело блока в соответствующий компилятор:

- `lang="scss"` и `lang="sass"` → SCSS compiler;
- `lang="less"` → Less compiler;
- `lang="styl"` и `lang="stylus"` → Stylus compiler.

Этот helper вызван из компиляторов Astro, Svelte и Vue. Поэтому зависимости внутри component styles теперь становятся частью графа анализа:

```astro
---
import Layout from "../layouts/Layout.astro";
---

<style lang="less">
  @import "@fontsource/lato/less/mixins.less";
  @import "./theme";
</style>
```

```svelte
<script>
  import Button from "./Button.svelte";
</script>

<style lang="stylus">
@require '~nib/lib/nib.styl'
@require './tokens'
</style>
```

До `6.24.0` такие связи приходилось либо дублировать обычными import'ами, либо гасить через настройки Knip. Теперь они извлекаются из самих компонентов, в том числе когда у Svelte/Vue нет script imports и весь полезный сигнал находится только в `<style>`.

Отдельный коммит [`3c9d4ad`](https://github.com/webpro-nl/knip/commit/3c9d4adf369f0064399d9da7b4f11f0467180bf5) меняет учёт самих compiler dependencies: `sass`, `less`, `stylus` и `@mdx-js/*` помечаются как referenced, когда включают встроенный компилятор Knip. Это закрывает ещё один класс ложных `unused dependencies`: пакет может не импортироваться в коде напрямую, но быть нужен именно для препроцессора.

## Stencil: SCSS из `styleUrl`, `styleUrls` и global styles

PR [#1846](https://github.com/webpro-nl/knip/pull/1846) расширяет Stencil plugin. Раньше Knip видел базовые entrypoints Stencil, но SCSS-файлы, подключённые через decorators и config, могли выглядеть как unused files. В `packages/knip/src/plugins/stencil/index.ts` добавлено сразу три механизма.

Первый — регистрация SCSS/Sass compiler, когда в проекте есть `@stencil/sass`:

```ts
if (hasDependency("@stencil/sass")) {
  for (const ext of [".scss", ".sass"]) {
    registerCompiler({ extension: ext, compiler: scss.compiler });
  }
}
```

Второй — AST visitor для `@Component(...)`. Он запоминает локальное имя `Component`, импортированное из `@stencil/core`, и извлекает пути из `styleUrl` и `styleUrls`:

```tsx
import { Component, h } from "@stencil/core";

@Component({ tag: "my-button", styleUrl: "my-button.scss" })
export class MyButton {
  render() {
    return (
      <button>
        <slot />
      </button>
    );
  }
}

@Component({ tag: "my-card", styleUrls: ["my-card.scss", "my-card-rtl.scss"] })
export class MyCard {}
```

Третий — разбор global styles из `stencil.config.ts`: обычного `globalStyle` и Stencil v5 `outputTargets` с `{ type: "global-style", input: "..." }`.

```ts
import { sass } from "@stencil/sass";

export const config = {
  globalStyle: "src/global-1.scss",
  outputTargets: [{ type: "global-style", input: "src/global-2.scss" }],
  plugins: [sass()],
};
```

Тест для fixture `plugins/stencil` специально оставляет `src/components/unused.scss` неиспользуемым и ожидает `files: 1`. То есть релиз не выключает отчёт по SCSS целиком: он отличает реально подключённые Stencil styles от соседних файлов, которые по-прежнему можно чистить.

## Меньше шума в GitHub Actions, JSDoc и module augmentation

Коммит [`aea7923`](https://github.com/webpro-nl/knip/commit/aea7923438f0a8f459458578526f358b02497f77) исправляет сценарий GitHub Actions с `working-directory`. Knip теперь передаёт resolver'ам manifest workspace'а, соответствующего директории шага, а yarn resolver проверяет scripts именно в этом manifest. Regression fixture добавляет workflow step:

```yaml
- name: Generate icons
  run: yarn icons
  working-directory: packages/app
```

и script в `packages/app/package.json`:

```json
{
  "scripts": {
    "icons": ""
  }
}
```

Раньше `yarn icons` мог трактоваться как binary `icons` и попадать в `unlisted binaries`, потому что resolver смотрел не на package.json рабочей директории. В `6.24.0` это считается script'ом workspace'а.

PR [#1844](https://github.com/webpro-nl/knip/pull/1844) сужает извлечение `import()` из JSDoc. Knip по-прежнему учитывает type expressions вроде `@type`, `@const` и `@member`:

```ts
/** @const {import('const-types').Setting} */
const setting = null;

/** @member {import('member-types').Field} */
const field = null;
```

но больше не превращает `import()` внутри `@example` в реальный unresolved import:

```ts
/**
 * @example
 *   { index: true, lazy: lazyComponent(() => import('./Foo'), 'Foo') }
 */
export const lazyComponent = (loader, name) => ({ loader, name });
```

Тест прямо проверяет `assert(!issues.unresolved['index.ts']?.['./Foo'])`.

Коммит [`7901abd`](https://github.com/webpro-nl/knip/commit/7901abd3c4b496f212445bff9768efc9548ded61) закрывает false positive по TypeScript module augmentation. Внутри `declare module './events.js'` типы `BaseEntity` и `EventEnvelope` резолвятся из exports аугментируемого модуля без явного import. Knip теперь собирает такие ссылки и регистрирует их как type-only imports:

```ts
declare module "./events.js" {
  type EntityWithMeta = BaseEntity & { meta: string };

  export interface EventBusEvents extends EventEnvelope {
    "entity:created": EntityWithMeta;
    "entity:updated": BaseEntity & { revision: number };
  }
}
```

В regression test оба типа из `events.ts` больше не должны попадать в `issues.types` как unused.

## Reporters, публичные типы и производительность компиляторов

PR [#1848](https://github.com/webpro-nl/knip/pull/1848) меняет вывод `default`/`symbols` и `compact` reporters для случая, когда в отчёт включена только одна группа issues. Раньше заголовок типа issue печатался только при нескольких группах; теперь `getIssueTypeTitle(reportType)` вызывается всегда.

Например, `knip --reporter compact --include dependencies --strict --no-config-hints` теперь ожидаемо печатает не только строку с пакетом, но и заголовок с количеством:

```txt
Unused dependencies (1)
package.json: unused-dep
```

Для `symbols` fixture с `--include files` аналогично добавлен заголовок:

```txt
Unused files (2)
packages/client/src/unused.ts
src/unused.ts
```

Коммит [`260f192`](https://github.com/webpro-nl/knip/commit/260f19230f42c9dceaac97d55bf34d17902c5b38) расширяет публичный type export из `packages/knip/src/types.ts`. Теперь из пакета доступны не только `Preprocessor`, `Reporter` и `ReporterOptions`, но и типы issues:

```ts
export type {
  Issue,
  IssueRecords,
  IssueType,
  Preprocessor,
  Reporter,
  ReporterOptions,
} from "knip";
```

Это полезно для кастомных reporters, preprocessors и внутренних интеграций, которым нужно типизировать входные данные отчёта без deep import'ов во внутренние файлы Knip.

Коммит [`bb0eeb6`](https://github.com/webpro-nl/knip/commit/bb0eeb6e33d050dab736134b5b692a3d6413f358) оптимизирует горячие места вокруг compilers. Из заметного: `runAsyncCompilers` больше не строит отдельный массив расширений и отфильтрованных путей, а проходит по `projectPaths` и проверяет `this.asyncCompilers.has(extname(filePath))`; компиляторы Less/SCSS/Stylus/MDX/Tailwind/Prisma добавили быстрые `text.includes(...)`-проверки перед regex-сканированием; несколько `matchAll` заменены на `RegExp.exec` с явным `lastIndex = 0`; `SourceFileManager.readFile` теперь корректно использует cached empty string, потому что проверяет `cachedSourceText !== undefined`, а не только `has`.

## Node 24+ для разработки Knip

Коммит [`d9ef038`](https://github.com/webpro-nl/knip/commit/d9ef038429bec06c37fdb520e5c7353c8cae6ce7) обновляет contributor workflow: Knip теперь предполагает Node.js v24+ или Bun и описывает запуск TypeScript CLI напрямую, без отдельного compile/watch шага.

```shell
node path/to/knip/packages/knip/src/cli.ts
```

Это изменение в первую очередь важно контрибьюторам Knip, а не пользователям CLI из npm-пакета. Для них релиз интереснее из-за новых компиляторов и сниженного шума в отчётах.

## Кому стоит обновиться

`6.24.0` особенно полезен проектам с Less/Stylus, Astro/Svelte/Vue component styles и Stencil с `@stencil/sass`: Knip теперь видит связи, которые раньше жили только в stylesheet syntax или decorators. Ещё одна группа — TypeScript-библиотеки с module augmentation и проекты с JSDoc-примерами: обновление уменьшает false positive'ы без необходимости добавлять локальные исключения.

Если вы строите интеграцию вокруг отчётов Knip, обратите внимание на экспорт `Issue`, `IssueRecords` и `IssueType`, а если используете `compact`/`symbols` reporter в CI — на более явные заголовки групп даже при одном типе issues.

Полный список изменений доступен в [GitHub Release 6.24.0](https://github.com/webpro-nl/knip/releases/tag/knip%406.24.0).
