[`antigravity-cli`](https://github.com/google-antigravity/antigravity-cli) выпустил минорную версию [`1.1.0`](https://github.com/google-antigravity/antigravity-cli/blob/main/CHANGELOG.md#110). Главная тема релиза — сделать агентные правки более обозримыми: дефолтное поведение теперь останавливается перед записью файлов, показывает интерактивный построчный diff и даёт принять или отклонить отдельные изменения до сохранения на диск.

Источник для обзора — `CHANGELOG.md` из GitHub API для [`google-antigravity/antigravity-cli`](https://github.com/google-antigravity/antigravity-cli/blob/main/CHANGELOG.md) и compare [`1.0.16...1.1.0`](https://github.com/google-antigravity/antigravity-cli/compare/1.0.16...1.1.0). В публичном репозитории diff между тегами содержит только обновление `CHANGELOG.md` и README-патч из [PR #326](https://github.com/google-antigravity/antigravity-cli/pull/326), который документирует запуск через `agy`; исходники самого TUI не опубликованы, поэтому конкретика ниже ограничена теми пользовательскими контрактами, которые Google зафиксировал в changelog.

## Default-поведение стало review-first

Самое важное изменение для ежедневной работы — новый `request-review` как дефолтное execution behavior. Перед операциями записи файлов агент больше не должен молча сохранять предложенный патч: CLI останавливается, показывает line-level diff preview и даёт пользователю принять или отклонить отдельные модификации.

В changelog отдельно указан shortcut `f`: он открывает интерактивный preview, где ревью идёт на уровне строк, а не только на уровне «разрешить весь tool call». Практический смысл такой: Antigravity CLI теперь ближе к code-review циклу внутри терминала. Если агент меняет несколько участков файла, пользователь может сохранить нужные куски и отсеять лишние до того, как изменения окажутся в рабочем дереве.

Для команды это меняет безопасный baseline: после обновления можно чаще оставаться в дефолтном режиме и не переключаться вручную в максимально осторожный workflow перед задачами, где агент затрагивает кодовую базу.

## Режимы execution mode теперь можно переключать явно

Публичным стал цикл режимов:

```text
default -> accept-edits -> plan
```

Changelog привязывает этот workflow к `shift+tab`: вместо набора старых slash-команд пользователь циклически меняет режим прямо из TUI. В релизе также закрепили `/plan` как новый префикс/режим планирования, заменив legacy-команду `/planning`, а `/fast` убрали.

Минимальные пользовательские сценарии теперь выглядят так:

```text
/plan Разложи миграцию на маленькие PR-готовые шаги
```

или, если режим выбирается при запуске:

```bash
agy --mode plan
agy --mode accept-edits
```

Важная деталь: changelog перечисляет именно `default`, `accept-edits` и `plan` как persistable значения Agent Mode. `request-review` при этом описан как дефолтное поведение записи файлов, а не как отдельное значение в списке для `/settings`.

## Agent Mode появился в `/settings` и синхронизируется в реальном времени

До 1.1.0 постоянный режим надо было задавать либо через `settings.json`, либо через `--mode` на старте. Теперь в панели `/settings` есть отдельная опция `Agent Mode`. Она сохраняет дефолтный execution mode из набора:

```text
default
accept-edits
plan
```

Changelog подчёркивает real-time synchronization: изменение в `/settings` начинает действовать сразу, без перезапуска CLI и без ручного редактирования конфигов. Это особенно полезно для команд, где разные задачи требуют разной степени автономности: например, обычную навигацию оставить в `default`, массовые безопасные изменения перевести в `accept-edits`, а архитектурные задачи начинать с `plan`.

## Создание новых файлов получило отдельный preview

Для `write_to_file` без overwrite появился dedicated `"Create file"` confirmation preview. Это отдельный сценарий от редактирования существующего файла: новый контент рендерится как addition-only diff.

Это мелочь только на первый взгляд. До такого preview создание файла часто выглядит менее проверяемым, чем патч к существующему файлу: нет старого контекста и проще пропустить лишний boilerplate. В 1.1.0 новый файл должен показываться как понятный diff из добавленных строк, то есть пользователь видит полный будущий файл до записи.

## Diff preview стал ближе к нормальному review-инструменту

Релиз отдельно улучшает rendering file-edit preview:

- diff считается на line-level, а не как грубый блок текста;
- вокруг изменений показываются `3` строки контекста;
- между участками есть hunk separators;
- inline preview ограничен по высоте и показывает подсказку о truncation, если diff не помещается;
- при выходе из diff view с ненаправленными комментариями появляется confirmation prompt.

Последний пункт важен для workflow с review comments: CLI теперь не должен позволять случайно потерять подготовленные замечания при закрытии diff view. В итоге `request-review` становится не просто «ещё одним подтверждением», а полноценным шагом ревью с контекстом, hunks и защитой от потери комментариев.

## Футеры панелей начали уважать `keybindings.json`

В `/tasks`, `/agents`, `/permissions`, `/mcp` и других панелях footer hints больше не должны быть зашиты статическими строками. Changelog говорит о переходе на centralized layout helpers, которые динамически учитывают global и local keybinding configuration из `keybindings.json`.

Практический эффект заметят пользователи, которые переназначают клавиши. Раньше UI мог показывать подсказку для дефолтного shortcut, хотя реальная команда уже находится на другой клавише. После 1.1.0 подсказки в футере должны соответствовать текущей конфигурации, а не документации по умолчанию.

## `/resume` стал устойчивее при переименовании разговоров

В picker `/resume` улучшили multiline conversation rename view. Во время активного переименования CLI теперь динамически подстраивает ширину input box и padding, а metadata columns (`workspace`, `steps`, `time`) выравнивает справа на верхней строке.

Это исправляет типичный TUI-дефект: при длинном имени разговора или широкой metadata-строке интерфейс начинает горизонтально скроллиться или прыгать при каждом символе. В 1.1.0 переименование должно оставаться стабильным, а метаданные — не вытеснять поле ввода.

## Исправления workspace, `/agents`, overlays и времени задач

В блоке fixes есть несколько конкретных пользовательских багов.

Во-первых, tool confirmation dialog теперь проверяет normalized file URI against active workspace directories. Это чинит ложное предупреждение `"Reason: outside workspace"` для валидных операций внутри workspace — например, при создании или чтении файла, который из-за URI-нормализации раньше ошибочно считался внешним.

Во-вторых, запуск CLI внутри dot-prefixed ancestors больше не должен ломать workspace initialization. Проблемный пример из changelog — путь вида:

```text
.parent_dir/project
```

Фильтр исключения теперь scoped строго к relative paths inside the workspace, а не отбрасывает dot-prefixed родительские директории целиком.

В `/agents` поправили две несостыковки документации интерфейса с реальными файлами:

- заголовок создания subagent больше не показывает `agent.json`, когда нужен `agent.md`;
- секция `"Create New Agents"` больше не отправляет пользователя в `~/.gemini/antigravity-cli/` для глобальных subagents, а указывает на `~/.gemini/config/` — именно эту директорию CLI сканирует при startup discovery.

Для полноэкранных overlay panels (`/changelog`, `/artifact`, `/bug`, `/settings`) исправили лишние statusline hints: `? for shortcuts` и дублирующий `Esc to cancel` больше не должны появляться внутри overlay, потому что CLI теперь корректно отслеживает overlay panel states.

Наконец, в `/tasks` и task detail views timestamps фоновых задач, инициированных агентом, переводятся из UTC в local timezone. Это убирает рассинхрон между временем в общем списке задач, деталях и локальным ожиданием пользователя.

## README: команда запуска теперь явно задокументирована

Единственный кодово-документационный PR между `1.0.16` и `1.1.0` — [#326](https://github.com/google-antigravity/antigravity-cli/pull/326). Он добавил в README секцию Usage с командой запуска после установки:

```bash
agy
```

Это не новая CLI-функция, но полезная правка для onboarding: install-команды уже были в README, а прямого «что запустить после установки» не хватало.

## Итог

Antigravity CLI 1.1.0 — минорный релиз про контроль над агентными изменениями. `request-review` по умолчанию, addition-only preview для новых файлов и более аккуратный diff view уменьшают риск случайно принять плохой патч. `/settings` с Agent Mode и публичный цикл `default -> accept-edits -> plan` делают режимы работы видимыми, а не спрятанными в `settings.json` и startup-флагах. Остальные изменения закрывают TUI-шероховатости: корректные keybinding hints, стабильное переименование в `/resume`, правильные workspace checks, точные пути для subagents и локальное время в `/tasks`.
