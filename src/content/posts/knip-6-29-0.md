Вышел минорный релиз [Knip 6.29.0](https://github.com/webpro-nl/knip/releases/tag/knip%406.29.0) — инструмента для поиска неиспользуемых файлов, зависимостей и exports в JavaScript- и TypeScript-проектах. В этой версии Knip начинает видеть импорты из файлов TypeScript Render Extensions (`.tsrx`), перестаёт обходить символьные ссылки при асинхронном globbing и получает короткий флаг `-u` для вывода длительности работы.

Источники: [GitHub Release](https://github.com/webpro-nl/knip/releases/tag/knip%406.29.0), [compare 6.28.0…6.29.0](https://github.com/webpro-nl/knip/compare/knip%406.28.0...knip%406.29.0), коммиты [glob](https://github.com/webpro-nl/knip/commit/b7d5ce2f545d91953a97733aa1567e8728b20644), [TSRX](https://github.com/webpro-nl/knip/commit/db05e0142a41fad2ad45d62ba5106e0d97c92b45) и [`-u`](https://github.com/webpro-nl/knip/commit/932b9262af275d209923a5efab1b6f23067bb006), а также [PR #1900](https://github.com/webpro-nl/knip/pull/1900). Версия `6.29.0` соответствует minor-границе semver, поэтому это самостоятельная статья; `featured` оставлен `false`.

## Встроенный компилятор для `.tsrx`

Knip добавил `.tsrx` в список расширений встроенных compiler'ов. Новый обработчик включается автоматически, когда в проекте найдена одна из зависимостей `@tsrx/core`, `@tsrx/react`, `@tsrx/preact`, `@tsrx/vue`, `@tsrx/solid` или `@tsrx/ripple`.

Это не полноценный компилятор TSRX: обработчик `collectImports` извлекает из текста только операторы `import` и передаёт их в граф модулей. Поэтому зависимости и локальные файлы, подключённые из компонента, считаются используемыми. Например, в таком файле Knip увидит связь с `octane` и `./format.ts`:

```ts
// greeting.tsrx
import { onMount } from "octane";
import { formatName } from "./format.ts";

type GreetingProps = { name: string };

export function Greeting(props: GreetingProps) @{
  onMount(() => {});
  const label = formatName(props.name);
  <h1>{label}</h1>
}
```

У ограничения есть практическое следствие: встроенный обработчик не восстанавливает exports из синтаксиса `.tsrx`, поэтому проверка неиспользуемых exports внутри таких файлов не появляется автоматически. В добавленном тесте `unused.tsrx` по-прежнему попадает в `issues.files`, а документированный путь для более детального анализа — заменить встроенный обработчик compiler'ом соответствующего фреймворка в конфигурации Knip.

## Async glob больше не заходит в симлинки

В асинхронной реализации `glob()` Knip теперь передаёт tinyglobby `followSymbolicLinks: false`; синхронный вариант уже работал так же. Изменение особенно заметно в монорепозиториях с pnpm: относительные workspace-паттерны наподобие `../../packages/*` раньше могли пройти через симлинк `node_modules` в pnpm store и обойти огромное дерево зависимостей.

В описании [PR #1900](https://github.com/webpro-nl/knip/pull/1900) приведён замер конкретного workspace: обход с `followSymbolicLinks: true` возвращал около 133 000 путей, а с `false` — около 800; время запуска Knip уменьшилось примерно со 106 до 7 секунд при идентичном отчёте. Это данные автора PR для его репозитория, а не универсальная гарантия производительности, но они объясняют цель правки.

Добавленные fixtures фиксируют две границы поведения:

- симлинк на используемый файл внутри проекта не должен создавать ложный `unused files`;
- симлинк, ведущий за пределы project scope, не добавляет внешний файл в набор обработанных путей.

Пользователям не нужно менять `knip.json`: флаг задан внутри `packages/knip/src/util/glob-core.ts`. После обновления стоит повторно запустить Knip в workspace с симлинками и сверить отчёт, если ранее он намеренно опирался на обход символьных ссылок.

## `-u` как короткая форма `--duration`

В парсере аргументов Node для опции `duration` добавлен `short: "u"`. Теперь две команды эквивалентны:

```bash
pnpm knip --duration
pnpm knip -u
```

Флаг относится к диагностике производительности: он включает вывод длительности работы. Это исправление именно shortcut — поведение `--duration` и формат его результата не менялись.

## Обновление

```bash
pnpm add -D knip@6.29.0
pnpm knip
```

Проектам с `.tsrx` стоит после обновления убрать из ручной конфигурации только те entries или ignores, которые были нужны исключительно для импортов из этих файлов. В крупных pnpm-монорепозиториях полезно также сравнить время и результаты первого запуска: новый async glob перестаёт раскрывать зависимости через симлинки, но нормальные исходные пути workspace Knip предварительно резолвит через `realpathSync`.
